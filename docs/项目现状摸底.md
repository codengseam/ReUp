## 一、项目背景与定位

### 1.1 产品定位

ReUp 是一款面向职场人士的 **RAG 顾问智能体**，以"资深 HR + 总裁"双视角提供职场建议。本质是一个 **RAG 实践项目**，用检索增强生成技术解决三类职场问题：

| 场景 | 优先级 | 核心价值 |
|------|--------|----------|
| 简历优化 | P0 | 7 维诊断 + ATS + STAR 重写 + 三格式导出 |
| 求职面试 | P0 | 模拟面试 + 面经分析 + 反问训练 |
| 晋升准备 | P1（附带） | 8 个 Skill 知识库 + 结构化问答 |

### 1.2 技术栈

| 层 | 技术 | 备注 |
|----|------|------|
| 框架 | Next.js 16 (App Router) + React 19 + TS5 strict | 单仓单服务 |
| 样式 | Tailwind 4 + shadcn/ui (Radix) | 主色 `#10b981` emerald |
| LLM | DashScope (Qwen) OpenAI 兼容 | `DASHSCOPE_API_KEY` |
| Embedding | BGE-M3 本地（1024 维，608 向量） | 预打包 `data/skill-vectors.json` |
| Rerank | BGE-reranker-v2-m3 本地（懒加载） | ⚠️ 已实现但生产链路未调用 |
| RAG | 语义 + 稀疏 + HyDE → 加权融合 → Top-K=5 | 自研 `src/lib/rag/` |
| 数据库 | Prisma (4 模型) + 原生 better-sqlite3 (7 表) | ⚠️ 双层并存 |
| 测试 | Vitest 4 | ⚠️ 覆盖率未强制 |
| 部署 | Docker → 魔搭空间 | 端口 7860，持久化 `/mnt/workspace` |

### 1.3 知识库

- **8 个 Skill**：晋升底层逻辑、晋升三大原则、能力三种境界、P8 领域专家、亮点挖掘、反问框架、盲区导航、能力模型对齐
- **书源**：`大厂晋升指南`(22 篇) + `面试现场`(32 篇) = 54 个 md → 608 chunks → 18MB 向量库
- **接入确认**：✅ 书源已真实 chunk + embed 进 `skill-vectors.json`，运行时懒加载，非"只是文件"

## 五、可实施方案（分阶段）

### Phase 0：阻断修复（最高优先级）

**目标**：消除 6 个 P0，让系统"真的能用"。

| 任务 | 模块 | 验收 |
|------|------|------|
| Prisma 表创建（entrypoint 加 `db push` 或补 migrations） | 部署 | 魔搭重启后 analytics/interview/offer/review 数据仍在 |
| 接通 BGE-reranker（去 `skipRerank:true`，over-fetch→cross-encoder→topK） | RAG | reranker 真实加载日志；检索精度提升 |
| 挂载 `<Toaster richColors />` 到 `layout.tsx` | 前端 | Admin 操作有 toast 反馈 |
| 流式首帧 HTML 消毒（同步引入 dompurify 或首帧纯文本占位） | 前端 | 首 token 无未消毒 HTML 注入 |
| 双 lockfile 治理（删 `package-lock.json`，CI 校验） | 工程 | 仅 `pnpm-lock.yaml` |
| `pdfkit` 从 devDeps 移入 deps | 工程 | `pnpm install --prod` 后 PDF 导出正常 |

### Phase 1：代码树收口

**目标**：消除双份代码，建立单一真相源。

| 任务 | 验收 |
|------|------|
| eslint R2 由 warn 升 error | 54 个 legacy import 暴露 |
| 按模块分批迁移（先 `lib/{offer,review,jd}` 零风险删，再 `lib/rag` 合并 `server/rag`，最后 `lib/resume`） | `pnpm ts-check && pnpm lint && pnpm test` 全绿 |
| `lib/` 副本改为纯 re-export shim 或删除 | lib/ 仅留 re-export |
| 双 DB 层合并（Prisma 与原生 better-sqlite3 界定边界或统一） | 单一 DB 访问层 |

### Phase 2：RAG RCR 三环节深化

**目标**：检索召回率/精确率、引用忠实度、响应可信度达到专业 RAG 标准。

**Retrieval（检索）**
- 激活 sparse_vector 为真 BM25 / 词法路径
- docId 去重替换 content 前 100 字
- min-max 归一化修复（单结果保持原分）
- cache hash 升 sha256，纳入完整历史

**Citation（引用）**
- citation id 绑 docId（跨轮稳定）
- 答案中 `[n]` 结构化校验：越界/孤立引用标记或截断

**Response（响应）**
- 幻觉检查改 fail-closed，低分阻断重生成
- 置信度解耦 hotQuery（hotQuery 只加权召回，不定置信）
- 接入 `eval/hallucination-detector` 双维度
- assessConfidence 基于真实召回质量

### Phase 3：测试与质量门禁

**目标**：核心链路有回归保护，覆盖率强制 80%。

| 任务 | 验收 |

**目标**：冷启动可控、健康检查可靠、回滚可控。

## 六、建议方案（推荐路径）

### 6.1 推荐执行顺序

```
Phase 0 (阻断修复)  ──必须先做──▶  Phase 1 (代码收口)
                                          │
                                          ▼
                                    Phase 2 (RAG RCR)  ──并行──▶  Phase 4 (魔搭优化)
                                          │
                                          ▼
                                    Phase 3 (测试门禁)
                                          │
                                          ▼
                                    Phase 5 (前端 + 外链)
```

- **Phase 0 → 1 必须串行**：不修 P0 就优化 RAG 是空中楼阁；不收口代码树就写测试会翻倍维护
- **Phase 2 与 4 可并行**：RAG 深化与魔搭部署优化无依赖
- **Phase 3 在 2 之后**：RAG 改完才能写稳定的回归基线
- **Phase 5 最后**：前端优化依赖后端契约稳定

### 6.2 关键设计决策

| 决策点 | 推荐 | 理由 |
|--------|------|------|
| 双份代码治理 | **激进收口**（eslint R2 升 error） | 一次性消除双份维护，避免长期漂移 |
| RCR 优化目标 | **Retrieval-Citation-Response 三环节** | 直接对应 RAG 核心质量，可度量 |
| 书籍跳转 | **先做外链**（YAGNI） | 外链足够，站内渲染页是过度设计 |
| 全局状态库 | **不引入**（localStorage 够用） | 当前规模无需 Zustand/Redux |
| 简历模块 | **仅修双份代码，不重写** | 已工业级，重写无收益 |
| 8 个 Skill | **不动内容** | 质量已高，仅优化接入 |

### 6.3 不做的事（YAGNI）

- ❌ 不新增功能（JD 独立页、Offer 自动带入属于"新增"非"优化"）
- ❌ 不做站内书籍渲染页（外链足够）
- ❌ 不引入全局状态库
- ❌ 不重写简历模块
- ❌ 不动 8 个 Skill 内容

| 三页面入口可见性 | 隐形 | ✅ 首页卡片入口 |

### 7.2 质性提升

| 维度 | 提升点 |
|------|--------|
| **专业度** | RAG 系统从"能跑"到"可度量、可评估、可回归"——RCR 三环节有量化指标 |

| **安全性** | 消除 XSS 窗口、PDF 导出 prod 可用、SSRF 收紧 |
| **用户体验** | 三场景入口可见、操作有反馈、长对话流畅、citation 可溯源到原书 |
| **可观测性** | 检索各阶段时延/命中率/模型加载可追踪，日志全结构化 |
| **部署可控** | 镜像版本化、健康检查覆盖 DB、回滚可控 |
