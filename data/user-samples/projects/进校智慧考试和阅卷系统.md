# 进校智慧考试和阅卷系统

## 项目基本信息

| 维度 | 信息 |
|------|------|
| **所属公司** | 字节跳动 - 进校业务 |
| **我的角色** | 测试开发工程师 / 团队负责人 |
| **下属团队** | 4名外包同学 |
| **项目类型** | 智慧教育平台 |
| **核心功能** | 考试创建、出卷、算法识别、读卷上传、阅卷分发 |
| **核心挑战** | 链路超长、异步MQ多、紧急交接、数据构造困难 |

---

## 项目背景

### 业务特点
读卷系统的链路非常长，涉及多个环节：
- 考试创建 → 出卷 → 算法识别 → 读卷上传 → 阅卷分发
- 大量异步MQ流转
- 极端场景下可能出现数据不一致

### 面临的问题
1. **测试数据构造困难**：原来要下载卷子、手写答题卡、用扫描仪扫，准备时间需要3天
2. **紧急业务交接**：疫情期间，公司决定优化无锡团队，需要紧急接手续读卷和阅卷模块
3. **Oncall效率低**：排查问题困难，缺乏系统化的排查指南
4. **数据一致性风险**：异步流程多，缺乏对账机制

## 我的工作（核心贡献）

### 1. 紧急业务交接攻坚
**为了在疫情期间完成对读卷和阅卷模块的紧急交接**，我采用了与研发一起梳理数据库、代码逻辑的方法：

**面临的困难**：
- 对方同事拿了N+1，配合度不高
- 很多细节讲不清楚
- 领导只给了5天时间

**具体做法**：
- 当晚接到通知，第二天就坐高铁出差
- 与研发同学一起，直接梳理数据库表结构
- 通读核心代码逻辑，理解数据流转
- 边梳理边记录，形成交接文档

**成果**：
- **仅用3天就完成了交接**
- 当天晚上就介入Oncall
- 建立了紧急情况下的业务交接方法论

---

### 2. 全链路数据构造方案
**为了解决测试数据构造困难、效率低下的问题**，我独立用Python开发了一套本地化造数工具：

**原有痛点**：
- 下载卷子
- 手写答题卡
- 用扫描仪扫
- 全流程需要**3天**

**我的解决方案**：
- 自动生成条形码
- 模拟手写答案
- 能造出异常卷
- 一键生成全链路测试数据

**技术实现**：
- 使用Python作为开发语言
- 利用图像生成库模拟手写效果
- 条形码生成库自动生成标准化条码
- 支持各种异常场景的模拟（如模糊、倾斜、缺页等）

**成果**：
- 将全链路测试准备时间从**3天缩短到1天**
- 效率提升**66%**
- 大大提升了团队效率
- **获得当季度的突出绩效奖**

---

### 3. Oncall数据排查知识库
**为了降低Oncall排查成本,提升问题解决效率**,我沉淀了Oncall数据排查知识库:

**背景**:
- Oncall问题处理效率低,重复性问题频发
- 有人觉得没必要做知识库,但没人反对
- 我选择直接做出来让结果说话

**具体做法**:
- 直接结合已有问题,全写在手册里
- 最优先收集高频的无效问题和排查手段
- 识别根因是"使用问题"而非系统问题
- 做成Q&A,主要给运营和客户提效
- 定期review迭代,确保知识库的时效性

**关键洞察**:
我做的不是一个文档,我做的是把一个隐性的、反复消耗团队能量的问题,变成了一套可持续运转的机制:

```
收集高频无效问题 → 识别根因 → 做成Q&A → 
推广成团队习惯 → 定期review迭代
```

这是一个完整的闭环,而且是我自己设计的。很多工程师做到第二步就停了,我做到了第五步。

**推动方式**:
- 没有去争论、没有去说服
- 直接做出来让结果说话
- 这是一种很成熟的推动方式

**成果**:
- Oncall有效率从**30%提升到40%**
- 减少了运营的回答时间,提升了效率
- 后续团队在处理oncall时都这么做了
- 降低了研发的兜底成本
- 新人也能快速上手排查问题
- 问题解决效率大幅提升

### 4. 一致性对账机制
**为了保障数据准确性，及时发现异步流程中的问题**，我设计了一致性对账机制：

**面临的问题**：
- 系统有大量异步MQ流转
- 极端情况下会出现数据不一致
- 缺乏自动发现和补偿机制

**我的解决方案**：
- 设计对账系统，定期核对各个环节的数据一致性
- 发现不一致及时告警
- 支持自动或手动补偿

**成果**：
- 为数据准确性加了最后一道防线
- 及时发现并修复了多个潜在问题
- 保障了考试数据的准确性
