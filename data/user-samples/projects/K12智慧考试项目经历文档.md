## 一、项目背景和挑战

### 1.1 项目概述

智慧考试和阅卷系统是字节跳动K12教育业务的核心产品之一，为全国中小学生提供在线考试、智能阅卷、学情分析等服务。系统涉及考试管理、试卷批改、成绩统计、数据报表等多个核心模块，日均处理百万级考试数据。

### 1.2 面临的挑战

**挑战一：紧急业务交接**
- 原团队因组织调整被优化，需要在极短时间内完成核心模块交接
- 业务知识断层风险高，文档不完善，核心逻辑依赖口头传承
- 交接期间业务不能中断，需保证线上服务稳定性

**挑战二：疫情期间异地出差**
- 疫情期间需要往返北京-武汉两地，协调团队远程办公
- 需要在不确定的环境下保证项目进度和质量

**挑战三：测试效率低下**
- 缺乏有效的造数工具，测试数据准备周期长
- 自动化覆盖率低，回归测试依赖人工
- Oncall问题有效率低，重复性问题频发

**挑战四：数据一致性问题**
- 系统涉及多个微服务，消息队列异步处理导致数据不一致
- 缺乏有效的对账机制，问题发现滞后

### 2.5 MQ消息对账机制设计（STAR）

#### Situation（背景）
智慧考试系统采用微服务架构，多个服务间通过消息队列（MQ）异步通信。存在以下问题：
- 消息丢失或重复消费导致数据不一致
- 问题发现滞后，影响业务准确性
- 缺乏有效的对账机制，问题排查困难

#### Task（任务）
- 设计MQ消息对账机制，保障数据一致性
- 覆盖百万级数据校验
- 修复历史一致性问题

#### Action（行动）

**1. 问题分析**
识别出以下一致性风险场景：
- 消息发送成功但消费失败
- 消息重复消费导致数据重复
- 消息顺序错误导致状态异常
- 网络超时导致消息丢失

**2. 对账机制设计**

**核心代码实现**
```python
class MQReconciliation:
    """MQ消息对账服务"""
    
    def __init__(self):
        self.producer_db = Database('producer_db')
        self.consumer_db = Database('consumer_db')
        self.diff_db = Database('diff_db')
    
    def reconcile_messages(self, start_time, end_time):
        """
        对账核心逻辑
        1. 查询生产者发送的消息
        2. 查询消费者消费的消息
        3. 比对差异并记录
        """
        # 查询发送消息
        sent_messages = self.producer_db.query(
            "SELECT * FROM message_send_log "
            "WHERE send_time BETWEEN %s AND %s",
            (start_time, end_time)
        )
        
        # 查询消费消息
        consumed_messages = self.consumer_db.query(
            "SELECT * FROM message_consume_log "
            "WHERE consume_time BETWEEN %s AND %s",
            (start_time, end_time)
        )
        
        # 构建消息索引
        sent_index = {msg['message_id']: msg for msg in sent_messages}
        consumed_index = {msg['message_id']: msg for msg in consumed_messages}
        
        # 检测差异
        differences = []
        
        # 1. 已发送但未消费的消息
        for msg_id, msg in sent_index.items():
            if msg_id not in consumed_index:
                differences.append({
                    'message_id': msg_id,
                    'type': 'NOT_CONSUMED',
                    'detail': msg
                })
        
        # 2. 已消费但未发送的消息（异常情况）
        for msg_id, msg in consumed_index.items():
            if msg_id not in sent_index:
                differences.append({
                    'message_id': msg_id,
                    'type': 'NOT_SENT',
                    'detail': msg
                })
        
        # 3. 消费状态异常的消息
        for msg_id, msg in consumed_index.items():
            if msg_id in sent_index:
                sent_msg = sent_index[msg_id]
                if sent_msg['status'] != msg['status']:
                    differences.append({
                        'message_id': msg_id,
                        'type': 'STATUS_MISMATCH',
                        'detail': {
                            'sent': sent_msg,
                            'consumed': msg
                        }
                    })
        
        # 记录差异
        if differences:
            self.diff_db.batch_insert('message_diff_log', differences)
        
        return differences
    
    def auto_fix_differences(self, differences):
        """
        自动修复差异
        """
        for diff in differences:
            if diff['type'] == 'NOT_CONSUMED':
                # 重新发送消息
                self.resend_message(diff['message_id'])
            elif diff['type'] == 'STATUS_MISMATCH':
                # 修正状态
                self.fix_status(diff['message_id'])
```

### 3.3 MQ消息对账机制设计

#### 面试官可能问的问题
1. "请介绍一下MQ消息对账机制的设计思路"
2. "如何保证消息不丢失、不重复"
3. "对账机制的性能如何保证"

#### 回答要点

**问题背景**
"智慧考试系统采用微服务架构，多个服务间通过MQ异步通信。我们发现存在消息丢失、重复消费等问题，导致数据不一致。最严重的一次，导致某次考试成绩统计错误，影响了数千名学生。"

**设计思路**
"对账机制的核心思路是：记录每条消息的发送和消费状态，定期比对差异。

第一，**消息记录**。生产者发送消息时，写入消息发送表。消费者消费消息时，写入消息消费表。

第二，**差异检测**。对账任务定期比对发送表和消费表，检测三类差异：
- 已发送但未消费的消息
- 已消费但未发送的消息（异常情况）
- 状态不一致的消息

第三，**自动修复**。对于常见差异场景，支持自动修复。例如，未消费的消息自动重发。

第四，**监控告警**。建立对账健康度监控，异常情况及时告警。"

**性能优化**
"对账机制的性能优化措施包括：

1. **增量对账**：每小时对账最近1小时的消息，减少数据量
2. **索引优化**：为消息表建立索引，提高查询效率
3. **批量处理**：采用批量查询和插入，减少数据库交互
4. **异步处理**：对账任务异步执行，不影响业务主流程"

**成果数据**
"对账机制上线后，覆盖百万级数据校验，发现并修复一致性问题10+个。数据一致率从99.5%提升至99.99%。问题定位时间从小时级降至分钟级。"

#### 可追问的亮点
- 如何处理消息顺序问题？
- 对账失败如何处理？
- 如何保证对账机制本身的高可用？
