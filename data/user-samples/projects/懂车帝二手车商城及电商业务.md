## 一、项目背景和业务架构（STAR - Situation）

### 1.1 业务线概览

我负责的电商业务线包括：

| 业务线 | 业务描述 | 用户规模 | 特点 |

| **新车电商** | 新车在线预订和购车服务 | DAU 30万+ | 与4S店深度合作 |
| **懂车购** | 汽车导购和线索转化平台 | MAU 200万+ | 线索分发为核心 |
| **二手车小程序** | 抖音生态内的二手车交易入口 | DAU 20万+ | 依托抖音流量 |

### 1.2 O2O业务模式特点

二手车电商是典型的O2O（Online to Offline）业务，与传统电商有本质区别：

```
┌─────────────────────────────────────────────────────────────────┐
│                    二手车O2O交易链路                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  线上环节                          线下环节                       │
│  ────────                          ────────                       │
│                                                                 │
│  用户浏览 ──→ 线索留资 ──→ 预约看车 ──→ 到店体验 ──→ 签约成交    │
│     │           │           │           │           │           │
│     ↓           ↓           ↓           ↓           ↓           │
│   曝光量      线索量      转化率      到店率      成交率          │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                    资金流转环节                           │    │
│  │  小订(意向金) → 大订(定金) → 尾款 → 金融贷款 → 过户费用   │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**O2O业务的核心挑战**：

1. **链路超长**：从线索到成交，涉及20+个业务节点
2. **线上线下耦合**：线上系统依赖线下门店配合
3. **资金敏感**：涉及真实资金流转，单笔金额可达数十万
4. **场景复杂**：退订、换车、金融方案变更等异常场景多

## 核心接口设计

### QueryCardList（列表查询）
- 功能：导购卡片列表查询，包含车源详情、动态价格、优惠活动、营销标签等
- 特性：支持MID路由、并行聚合、容灾降级
- 容灾：MustRecover → Main Process → AutoRecover → Record

### QueryCardDetail（商详查询）
- 功能：商品详情聚合，包含基本信息、SKU详情、库存状态、金融标签等
- 特性：缓存降级、参数补齐、登录态解析
- 缓存：Redis热点缓存（5小时），支持降级回放

### 报价单接口
- 功能：多中心聚合，价格计算、优惠试算
- 特性：BFF负责参数解析，价格中心负责核心计算

### 预下单接口
- 功能：风控状态校验、用户白名单校验
- 特性：支持车源单和券商品两种模式

## 容灾降级方案

### 多级缓存架构
- L1（本地LRU）：存储高频静态配置，如MID路由配置
- L2（Redis）：存储热点动态数据，支持Snappy压缩
- L3（Abase）：全量数据存储，PB级容量

### 列表页容灾
- 存储：Redis，支持ZSet高热队列
- 策略：TTL自动过期 + 异步回源
- 降级：故障时回放历史成功数据

### 商详页容灾
- 存储：Abase，基于RocksDB持久化
- 策略：主动构建 + 版本化更新
- 降级：多副本异地多活，手动切换缓存

### 降级流程
1. MustRecover（主动降级）：依据配置直接读缓存返回
2. Main Process（执行业务）：正常业务流程
3. AutoRecover（自动降级）：主流程失败时自动读缓存
4. Record（数据记录）：成功时异步写入缓存

**典型案例**：

> **案例：订单状态同步延迟问题**
> 
> 在门店调研时发现，销售在系统中确认订单后，用户端状态更新存在延迟，导致用户频繁询问"订单状态怎么还没变"。
> 
> **问题原因**：状态同步依赖MQ异步处理，高峰期存在延迟。
> 
> **解决方案**：推动研发优化为同步+异步双写机制，状态更新延迟从5秒降低到1秒以内。
> 
> **效果**：用户投诉率下降30%，销售满意度显著提升。

### 3.4 资金安全白皮书

**背景（S）**：涉及真实资金流转,缺乏统一的资金安全标准,存在事故隐患

**核心动机**:
这不是接需求,是我主动从风险视角重新审视整个业务的结果。二手车交易涉及真实资金流转,单笔金额可达数十万。之前团队缺乏统一的资金安全标准,存在事故隐患。我主动梳理了所有资金环节,输出这份白皮书。

**任务（T）**：建立资金安全体系,防止资金事故

**行动（A）**：

1. **资金环节梳理**
   - 梳理从线索到交易完成的所有资金节点

2. **风险点识别**
