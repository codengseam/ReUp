# 规则：JD 结构化拆解

> 本文件定义 jd-decoder 第 1 步的执行细则。返回上层路由见 `../SKILL.md`。

JD 文本经本规则处理后，必须输出 8 个核心字段。每个字段含：提取规则、识别模式、模糊情况处理。

## 8 个核心字段总览

| 序号 | 字段 | 是否必填 | 默认值（无法识别时） |
|------|------|---------|---------------------|
| 1 | 职位名称 (title) | 必填 | "未知职位" |
| 2 | 部门 (department) | 选填 | 空 |
| 3 | 职级 (level) | 选填 | 空（交由 `jinsheng-san-yuanze` 推断段位） |
| 4 | 薪资范围 (salary) | 选填 | 空 |
| 5 | 硬性要求 (hardRequirements) | 必填 | 空数组 |
| 6 | 岗位职责 (responsibilities) | 必填 | 空数组 |
| 7 | 技能要求 (skills) | 必填 | 空数组 |
| 8 | 考察重点 (focusPoints) | 必填 | 空数组（交由 `./focus-points-inference.md` 填充） |

字段 8 的内容由 `./focus-points-inference.md` 二次填充，本规则只负责把候选线索喂给它。

---

## 字段 1：职位名称 (title)

### 提取规则
- 优先级 A：匹配"招聘/诚聘/急聘/招募 + 职位名"模式。
- 优先级 B：取 JD 正文第一段非噪音行（去除前缀编号、括号、方括号）。
- 优先级 C：匹配"职位/岗位/Position/Title + 冒号 + 名称"模式。

### 识别模式（正则骨架）
```
/(?:招聘|诚聘|急聘|招募)\s*[:：]?\s*([^，,\n、。]{2,20})/
/(?:职位|岗位|Position|Title)\s*[:：]\s*([^\n，,]{2,30})/i
```

### 模糊情况处理
- 职位名带城市/年限修饰（如"北京-高级前端工程师（3-5年）"）：剥离城市与年限括注，只保留核心职位名"高级前端工程师"，剥离部分回填到对应字段。
- 多个职位并列（如"前端/全栈工程师"）：保留原始复合名，不强行拆分。
- 全是噪音（如首行是"【字节跳动】"）：降级到优先级 B 的下一行。

---

## 字段 2：部门 (department)

### 提取规则
- 匹配"部门/团队/事业部/BU/Department/Team + 冒号 + 名称"。
- 匹配"XX 业务线 / XX 中台 / XX 实验室"作为部门别名。

### 识别模式
```
/(?:部门|团队|事业部|业务线|中台|实验室|BU|Department|Team)\s*[:：]\s*([^\n，,]{2,30})/i
```

### 模糊情况处理
- JD 标题已含部门（如"抖音电商-服务端开发"）：用"-"或"·"分隔，取前半段为部门，后半段为 title 补充。
- 无明确部门词但出现"隶属于 XX 中心"：取"XX 中心"为部门。
- 仍无法识别：留空，不猜测。

---

## 字段 3：职级 (level)

### 提取规则
- 直接匹配职级标识：P5/P6/P7/P8、T2.1/T3、L5/L6/L7、阿里/腾讯/字节/美团职级体系。
- 匹配职级描述词："初级/中级/高级/资深/专家/架构师/初级/Associate/Senior/Staff/Principal"。

### 识别模式
```
/\b([PTL]\d{1,2}(?:\.\d)?)\b/
/(资深|高级|中级|初级|专家|架构师|Associate|Senior|Staff|Principal)/
```

### 模糊情况处理
- 只给描述词无数字职级（如"资深工程师"）：保留描述词，数字段位留空，交由 `jinsheng-san-yuanze` 三原则映射推断段位（"主导优化"→价值原则→P7+；"独立负责"→主动原则→P5/P6）。
- 同时出现数字职级与描述词（如"P7 资深专家"）：两者都保留，以数字职级为准。
- 职级冲突（JD 同时写 P6 和 P7）：标注 `level_conflict: true`，取较高者并备注。

---

## 字段 4：薪资范围 (salary)

### 提取规则
- 薪资关键字（薪资/薪水/薪酬/工资/salary）+ 区间数字。
- 数字带 k/K 后缀（如 25k-40k），按 k×1000 换算。
- 带"薪/月薪/年薪/14薪/16薪"等修饰。

### 识别模式
```
/(?:薪资|薪水|薪酬|工资|salary)[^\n]{0,30}?(\d+)([kK]?)\s*[-~至到]\s*(\d+)([kK]?)/i
/(\d+)([kK])\s*[-~至到]\s*(\d+)([kK]?)/
```

### 模糊情况处理
- 防误判："3-5 年经验"不应被当薪资。强制要求：要么前有薪资关键字，要么至少一个数字带 k 后缀。
- 只有上限或下限（如"薪资 30k+"）：缺失端填 null，并标注 `open_ended: true`。
- 年薪/月薪混淆：出现"年薪/annum"标 `period: "year"`，否则默认 `period: "month"`。
- 多币种：出现"$"标 `currency: "USD"`，"RMB/元/￥"标 `currency: "CNY"`。

## 字段 5：硬性要求 (hardRequirements)

### 提取规则
拆为 4 个子类，每条标记 `priority: "must" | "preferred"`：

| 子类 | category | 识别关键字 | must 触发条件 |
|------|----------|-----------|--------------|
| 学历 | 学历 | 本科/硕士/博士/大专/Bachelor/Master/PhD | 出现"及以上/统招/全日制" |
| 年限 | 经验 | N年经验/N年以上/years experience | 出现"以上/至少" |
| 证书 | 证书 | CPA/PMP/CCIE/司法考试/驾照 | 出现"必须持有/必备" |
| 其他 | 其他 | 行业背景/语种/出差要求 | 出现"必须/硬性" |

### 识别模式
```
/(本科|硕士|博士|大专)(?:及以上)?/
/(\d+)\s*年(?:以上)?(?:工作)?经验/
/(CPA|PMP|CCIE|HCIE|软考|司法考试)/
```

### 模糊情况处理
- "优先/加分/prefer"修饰 → `priority: "preferred"`，不计入 must 红线。
- "统招本科"等限定：在 description 中保留"统招"字样，不丢失修饰。
- 学历与年限写在同一句（如"5 年经验本科以上"）：拆为两条独立 hardRequirement。
- 无明确年限但出现"应届/校招" → category=经验，description="应届"。

---

## 字段 6：岗位职责 (responsibilities)

### 提取规则
- 定位"岗位职责/工作内容/Job Responsibilities/What you'll do"段落到下一空段或下一小标题之间。
- 按列表项切分：编号项（1./2./、-、*、•）或换行分隔。
- 每条去除前缀符号，保留为独立字符串。

### 识别模式
```
/(?:岗位职责|工作内容|职责描述|Job Responsibilities|What you['']ll do)\s*[:：]?/
/^\s*(?:\d+[.、]|[•\-\*])\s*(.+)/
```

### 模糊情况处理
- 整段无列表标记的散文式职责：按句号或分号切分，过滤掉 < 8 字的碎片。
- 职责与要求混排（无小标题区分）：用动词启发式分类——以"负责/主导/参与/推动/设计/优化/保障"开头的归职责；以"精通/熟悉/具备/要求/持有"开头的归技能或硬性要求。
- 同一条职责含多个动作（"负责 X 并主导 Y"）：拆分为两条，避免聚合失真。

---

## 字段 7：技能要求 (skills)

### 提取规则
- 定位"任职要求/岗位要求/技能要求/Qualifications/Requirements"段落。
- 每条技能标注：`name`、`level`（精通/熟悉/了解）、`required`（true/false）。
- 掌握程度词映射：

| JD 原词 | level | 说明 |
|---------|-------|------|
| 精通/深入理解/源码级/专家 | 精通 | 对应 `nengli-sanzhong-jingjie` 的"精通"境界（能优化/创造新经验） |
| 熟悉/掌握/熟练/能独立 | 熟悉 | 对应"熟练"境界（最佳实践、高质高效） |
| 了解/听说过/加分 | 了解 | 对应"基础"境界（会做） |

### 识别模式
```
/(?:任职要求|岗位要求|技能要求|任职资格|Qualifications|Requirements)\s*[:：]?/
/(精通|深入理解|源码级|熟悉|掌握|熟练|了解|听说过)/
```

### 模糊情况处理
- 无程度词的裸技能（如"会 Python"）：默认 `level: "熟悉"`，`required: true`。
- "加分项/优先/plus"修饰 → `required: false`。
- 技能集合词（如"前端技术栈"）：不拆，保留为一条，但在 description 补注"需进一步展开"。
- 同一技能多次出现程度词冲突（"熟悉 Java，精通 JVM 调优"）：拆为两条技能（Java=熟悉，JVM 调优=精通），不取平均。

---

## 字段 8：考察重点 (focusPoints) —— 候选线索

### 提取规则
本规则只负责从字段 5/6/7 中提取**候选线索**，最终 3-5 个维度的筛选与排序交由 `./focus-points-inference.md`。

候选线索收集规则：
- 硬性要求中 category=技能 且 priority=must → 候选"技术深挖"。
- 岗位职责中含"带领/管理/主导" → 候选"团队管理"。
- 岗位职责/技能中含"高并发/大数据量/分布式/海量" → 候选"系统设计"。
- 岗位职责含"从 0 到 1/搭建/0-1" → 候选"0-1 建设能力"。
- 软性描述段（"抗压/沟通/拥抱变化"） → 候选"行为面试"。

### 模糊情况处理
- 候选线索 > 5 条：按出现频次 + must 优先级排序，取前 5。
- 候选线索 < 3 条：不强行凑数，标注 `focus_points_insufficient: true`，交由 `./focus-points-inference.md` 用职责段补足。

## 中英文 JD 识别差异

| 维度 | 中文 JD | 英文 JD |
|------|---------|---------|
| 段落标记 | "岗位职责/任职要求" | "Responsibilities/Requirements/Qualifications" |
| 列表前缀 | 1./一、/•/- | 1./-/•/a) |
| 职级体系 | P5-P8 / T2-T4 / L5-L8 | L3-L8 / IC3-IC7 / E3-E9 |
| 薪资单位 | k（千）/万/年 | k（千）/$/£/EUR |
| 程度词 | 精通/熟悉/了解 | Expert/Proficient/Familiar |
| 学历 | 本科/硕士/博士 | Bachelor/Master/PhD |
| 模糊词 | "及以上" | "or equivalent" |

### 处理策略
- 语言判定：中文字符占比 > 30% 判为中文 JD，否则英文 JD；混合 JD（如外企中文岗）按主语言处理，副语言字段做兼容映射。
- 英文程度词映射：Expert→精通，Proficient/Strong→熟悉，Familiar/Working knowledge→了解，Nice-to-have→`required: false`。
- "or equivalent" → 视为 `priority: "preferred"`，不作为 must 红线。
- 中英混排职级（如 "P7 / Senior Staff"）：两套都保留，level 字段取中文体系为主。

- 职级段位 → `jinsheng-san-yuanze`（见 `../SKILL.md` 编排表）
- 技能境界 → `nengli-sanzhong-jingjie`
