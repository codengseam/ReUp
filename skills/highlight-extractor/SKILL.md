# "详历"四维亮点挖掘法

## 概念 (Concept)

### I — 方法论骨架 (Interpretation)

简历不应是项目数量或技术名词的堆砌（那是无效的"干扰项"）。真正的亮点不是必须参与 10 亿级的大项目，而是你如何在一件小事中体现出超常的特质。
本方法提供了一个结构化的四维清单来榨取"平淡经历"中的亮点：
1. **价值**：从老板/客户/团队的视角看，你这件小事解决了谁的痛点？
2. **结果**：有没有客观的数据指标因为你的工作变好了（效率、质量、成本）？
3. **创新**：在做这件小事时，你引入了什么新工具、简化了什么老流程？
4. **动机**：你面临过什么挑战或压力？你是靠怎样的内在驱动力挺过来的？

## 解释 (Explanation)

### E — 可执行步骤 (Execution)

当 skill 被激活后, agent 应按以下步骤执行:

1. **输入平淡经历**
   - 让用户描述一件他认为最平淡或耗时最长的工作（如修bug、写报表）。
   - 完成标准: 获得一段不少于 50 字的客观流水账描述。
   - **[检查点]**: 必须暂停，等待用户提供经历后再进入下一步，不可替用户编造。

2. **价值与结果榨取（What & How Good）**
   - 追问用户：这件事如果没有你做，对团队或客户有什么损失？你做完后，有没有什么数字（如耗时、投诉率、代码量）变好了？
   - **价值挖掘话术**：
     ```
     【价值转换】"这件事如果没有你做，会怎样？谁会受影响？"
     【结果量化】"做完后，有什么数字变好了？效率提升了多少？错误减少了多少？"
     【全局视角】"这件事对团队/公司有什么贡献？解决了什么痛点？"
     ```
   - **量化指标参考**：
     | 工作类型 | 可量化指标 |
     |---------|-----------|
     | 修Bug | Bug修复数量、线上故障率下降、用户投诉减少 |
     | 写报表 | 报表生成时间、数据准确率、决策效率 |
     | 带新人 | 新人上手时间、新人独立产出时间 |
     | 代码重构 | 代码行数减少、性能提升、可维护性评分 |
   - 完成标准: 提取出 1 个能用数字量化或体现全局价值的结果。
   - **[检查点]**: 暂停等待用户回答。若用户确实无法提供数字，则走 fallback：引导用户回忆"同事/老板对此事的正面评价"。

3. **创新与动机榨取（How & Why）**
   - 追问用户：在重复做这件事的过程中，你是否优化了步骤？引入了脚本？面对这种枯燥，你当时最头疼的是什么，怎么克服的？
   - **创新挖掘话术**：
     ```
     【工具创新】"有没有写过脚本/工具来简化流程？"
     【流程优化】"有没有发现并改进过什么低效步骤？"
     【方法沉淀】"有没有总结出什么可复用的方法/模板？"
     ```
   - **动机挖掘话术**：
     ```
     【挑战识别】"做这件事时，最头疼的是什么？"
     【内在驱动】"是什么让你愿意坚持下去？"
     【超越本职】"有没有做过超出本职要求的努力？"
     ```
   - 完成标准: 提取出 1 个微创新细节或展现坚韧/精益求精动机的细节。
   - **[检查点]**: 暂停等待用户回答。

4. **生成高价值亮点句**
   - 舍弃原有的流水账，将上述提取的元素融合成 1-2 句符合"简历减法原则"的高密度亮点句。
   - **亮点句生成模板**：
     ```
     【价值+结果】"负责XX系统监控，为60%的线上错误成功报警，保障系统稳定性"
     【创新+结果】"开发自动化脚本，将报表生成时间从2小时缩短到10分钟"
     【动机+结果】"主动承担烂尾项目重构，在无额外资源情况下完成上线，系统性能提升50%"
     【四维融合】"主导XX模块开发，解决团队长期痛点，引入自动化测试将Bug率降低30%，沉淀为团队最佳实践"
     ```
   - 完成标准: 产出可以直接写入简历或面试口述的 STAR 结构高光短句。

### B — 边界 (Boundary) ★

#### 不要在以下情况使用此 skill

- 当用户的经历本身就涉嫌造假或虚构时（本技能是榨取真实的果汁，不是无中生有兑水）。
- 当用户的项目本身已经是极其复杂的架构设计且其苦恼在于"不知道如何讲清技术架构"时（此时需要梳理技术逻辑，而不是找价值动机）。

#### 作者在书中警告的失败模式

- **被动罗列干扰项**：简历上列的项目越多越好，堆的技术名词越多越好，反而掩盖了真正的亮点。必须做减法，只留经得起推敲的价值点。

#### 容易混淆的邻近方法论

- 夸大其词的简历包装（包装是无中生有，亮点挖掘是寻找未被察觉的真实价值）。

## 场景 (Scenario)

### A1 — 书中的应用 (Past Application)

#### 案例 1: 枯燥的日志监控工作
- **问题**: 负责日常且不起眼的日志监控系统，看似毫无技术难度。
- **方法论的使用**: 用"结果量化"和"价值转换"维度进行挖掘。
- **结论**: 关注该监控对整个线上稳定性的贡献度。
- **结果**: 提炼出亮点——"我负责的日志监控，为60%的线上错误成功报警。"

#### 案例 2: 团队打杂与带新人
- **问题**: 花了很多时间带新人，这在纯技术开发眼中可能是"没亮点"的非核心工作。
- **方法论的使用**: 用"价值转换"维度，从团队和效率的角度审视。
- **结论**: 发现带新人这件"杂事"极大提升了团队的生产力。
- **结果**: 提炼出亮点——"我做的新人培训文档，使新人一周即可开展项目工作。"

### A2 — 触发场景 (Future Trigger) ★

#### 用户会在什么情境下需要这个 skill?

1. 用户在写简历，觉得自己的经历太水、太日常，不知道怎么写出彩。
2. 面试官问"说说你在这家公司最大的贡献/亮点"，用户脑海中只有流水账。
3. 用户抱怨"我每天就是做些增删改查/修bug，没有挑战性"。

#### 语言信号 (用户的话里出现这些就应激活)

- "我的经历没有亮点怎么办？"
- "每天就是写业务代码，简历怎么包装？"
- "帮忙润色一下这段工作经历"
- "我觉得我做的东西太普通了"

#### 与相邻 skill 的区分

- 与 `competency-model-alignment` 的区别: 本 skill 是**微观的素材榨汁机**，专门从一件事中提取"事实层面的闪光点"；后者是**宏观的回答结构器**，用于展示潜力和动机。
