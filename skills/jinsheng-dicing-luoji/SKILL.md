# 晋升底层逻辑 (提前做下一级别的事)

## R — 原文 (Reading)

> 评委们会审查你做过的事情，看看是不是体现了下一级别需要的能力。这就是第一条晋升逻辑：在当前级别做下一级别事情的人，才有机会晋升。第二条晋升逻辑：只有把当前级别的事情做好了，才有机会晋升。
>
> — 李运华, 《大厂晋升指南》第4章

---

## I — 方法论骨架 (Interpretation)

晋升不是对你过去"苦劳"或"把本职工作做得很好"的奖励，而是公司对你**已经具备下一级别能力**的官方确认。
因此，晋升的底层逻辑包含两步：首先，在当前级别达到"精通"（不仅仅是熟练完成，而是能优化现有流程）；其次，在达到精通后，主动承担并完成下一级别要求的工作（如 P6 开始带小团队，P7 开始做跨团队架构设计）。只有拿到了下一级别工作的实际成果，评委才会认可你达到了晋升标准。

---

## A1 — 书中的应用 (Past Application)

### 案例 1: P6 晋升 P7 的困境
- **问题**: 很多 P6 员工把手头的开发任务完成得又快又好，绩效也不错，但连续几次申请 P7 晋升都被拒绝，感到十分委屈。
- **方法论的使用**: 运用底层逻辑进行对照。P7 的核心要求是"带过小项目或小团队（3-5人）"。
- **结论**: 该员工虽然 P6 的本职工作做到了极致，但没有提前去争取带项目的机会，没有下一级别能力的数据证明。
- **结果**: 指导该员工主动向主管申请带领接下来的一个业务小战役，拿到结果后再去答辩，成功晋升。

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. 刚拿到好绩效，自信满满准备晋升，想做提前规划时。
2. 连续多次晋升失败，感到委屈，认为评委不公平或有潜规则时。
3. 刚入职或刚升职，想知道接下来的1-2年该如何开展工作时。

### 语言信号 (用户的话里出现这些就应激活)

- "我年终拿了A，明年能晋升吗？"
- "我每天加班把活儿干得漂漂亮亮，为什么主管不提名我？"
- "我下次答辩应该重点突出什么？"
- "晋升 PX 需要什么硬性条件？"

### 与相邻 skill 的区分

- 与 `nengli-sanzhong-jingjie` 的区别: 本 skill 侧重于宏观路径规划（从当前到下一级）；而 `nengli-sanzhong-jingjie` 侧重于评估当前手头的工作到底处于什么水平（基础还是精通）。

## E — 可执行步骤 (Execution Agent 视角)

作为执行层 Agent，你的核心任务是进行深度诊断和提炼，**不要**在这个阶段操心输出格式和合规引用，那是审查层的事情。当 skill 被激活后, 请按以下步骤执行:

1. **确认当前问题是否仍有晋升通道**
   - 先询问用户：你所在公司/团队当前是否还有明确晋升通道、晋升周期和可争取的高一级任务？
   - **[检查点]**: 必须暂停，等待用户回答组织环境。
   - 判停条件：如果用户明确处于锁 HC、冻结晋升、组织剧烈收缩阶段，则终止本 skill，直接指出"当前主要矛盾不是晋升准备，而是机会不足"。
   - 完成标准: 判断这次讨论该聚焦"能力准备"，还是先处理"组织机会"问题。

2. **评估当前级别完成度**
   - 询问用户：你当前的职级是什么？日常核心工作职责有哪些？你认为自己目前对这些职责的掌握程度是"基础（会做）"、"熟练（做好）"还是"精通（能优化流程/创造新经验）"？
   - **当前级别评估表**：
     | 掌握程度 | 行为特征 | 是否具备晋升基础 |
     |---------|---------|----------------|
     | 基础 | 能独立完成，但效率一般 | 否，需先达到熟练 |
     | 熟练 | 高效完成，符合规范 | 否，需先达到精通 |
     | 精通 | 能优化流程，创造新经验 | 是，可以开始越级尝试 |
   - **[检查点]**: 暂停等待用户回答。
   - 收到回答后，判定用户是否真正在当前岗位达到了"精通"。
   - 完成标准: 明确用户的当前状态是"还在补课"、"熟练"还是"已精通"，并向用户输出诊断结果。

3. **对标下一级别要求**
   - 询问用户：你的下一个目标职级是什么？你了解公司对该职级的核心要求吗？
   - **[检查点]**: 暂停等待用户回答。
   - 若用户不了解，按行业惯例给出参考：
     | 职级 | 核心能力要求 | 典型工作范围 |
     |------|-------------|-------------|
     | 初级(P5) | 独立执行 | 独立完成模块开发 |
     | 高级(P6) | 独当一面 | 负责完整项目，解决复杂问题 |
     | 资深(P7) | 带小团队/项目 | 3-5人团队管理，跨团队协作 |
     | 专家(P8) | 领域影响力 | 多团队管理，技术规划，梯队建设 |
   - 完成标准: 向用户列出下一级别的 1-2 项核心能力要求。

4. **寻找越级尝试的机会**
   - 结合上两步的信息，引导用户思考：在当前团队业务中，有哪些属于"下一级别"的事情是你近期可以主动揽过来的？
   - **越级机会清单**：
     ```
     【P6→P7】主动带实习生/新人、主导小项目、跨团队协作、技术分享
     【P7→P8】培养P7骨干、跨团队技术规划、领域影响力建设
     【通用】主动承担烂摊子并成功翻身、主导技术重构、输出最佳实践文档
     ```
   - **[检查点]**: 暂停等待用户确认哪些机会真实存在，不要替用户虚构机会池。
   - 完成标准: 输出一个结构化的"越级打怪计划"（包含 1-3 项具体、可落地的越级行动建议）。

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 用户连当前级别的基础任务都无法按时保质完成时（此时教他"做下一级别的事"会导致本职工作崩盘）。
- 公司正在大规模裁员、锁 HC，完全没有晋升通道时（此时应转向面试/跳槽辅导）。

### 作者在书中警告的失败模式

- 误以为晋升后立刻就能拿到下一级别的高级任务：主管往往不敢把高级任务给刚晋升且还未达到新级别"精通"状态的人。
- 熬资历：以为"我干了3年 P6 就该升 P7"，年限从来不是晋升的核心标准。

### 作者的盲点 / 时代局限

- 大厂高速扩张期，只要有能力"越级揽活"总有事做。但在业务收缩期，可能根本没有"下一级别的项目"可供你练手。

### 容易混淆的邻近方法论

- 容易和 `nengli-sanzhong-jingjie` 混淆。能力三境界是在当前职责里评估深度，本 skill 是看你是否已经拿到了更高一级的任务与成果。
- 容易和 `jinsheng-san-yuanze` 混淆。三原则决定"接哪些活"，本 skill 决定"哪些活才算下一级别的证据"。
- 容易和 `p8-lingyu-zhuanjia` 混淆。本 skill 适用于普遍的晋升跃迁判断，而 `p8-lingyu-zhuanjia` 专注多团队/领域专家阶段的职责重构。
