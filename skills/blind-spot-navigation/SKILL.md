# 知识盲区降维与平移策略

## 概念 (Concept)

### I — 方法论骨架 (Interpretation)

在面试中遇到"真不会"的问题时，死撑硬编是大忌，直接说"不会"则等于放弃得分机会。
高手的做法是：
1. **坦诚边界**：明确说明这部分自己不熟悉/没做过。
2. **主动出击（平移/降维）**：
   - **变化范围（平移）**：我没用过 A 技术，但我精通它的同类竞品 B 技术，它们的底层逻辑是相似的。
   - **变化维度（降维/升维）**：我不懂这个开源库的底层 C++ 源码（内部实现），但我非常清楚在何种高并发业务场景下该如何对它进行调优和选型（外部应用）。
通过这种方式，即使你没答出原问题，依然展示了你强大的学习潜力、业务 sense 和解决同类问题的能力。

## 解释 (Explanation)

### E — 可执行步骤 (Execution)

当 skill 被激活后, agent 应按以下步骤执行:

1. **定位盲区属性**
   - 让用户给出那道他"不会"的面试题。
   - 完成标准: 判断该问题是属于"某个具体的技术栈/工具"（适合平移），还是属于"某项技术的底层原理/宏观架构"（适合变维）。
   - **[检查点]**: 暂停，等待用户给出具体问题后再分析。

2. **建立坦诚防线**
   - 指导用户用 1-2 句话大方承认盲区，但绝不流露慌张。
   - **坦诚话术模板**：
     ```
     【技术栈盲区】"这个特定的框架我确实没有在生产环境使用过，不过..."
     【底层原理盲区】"这个底层实现我没有深入研究过源码，但..."
     【业务场景盲区】"这个业务场景我之前没有直接接触过，但类似的..."
     ```
   - 完成标准: 产出如"这个特定的库我确实没在生产环境用过，不过..."的破冰句。
   - **[检查点]**: 向用户确认该破冰句是否符合其日常语气，暂停等待反馈。

3. **设计降维/平移话术**
   - 引导用户寻找替代品。
   - **策略选择对照表**：
     | 盲区类型 | 推荐策略 | 话术模板 |
     |---------|---------|---------|
     | 没用过某技术栈 | 平移 | "我没有用过A，但我用过类似的B，它们的底层原理是..." |
     | 不懂底层源码 | 降维 | "源码层面我没有深究，但在业务应用层面，我处理过..." |
     | 没有某场景经验 | 升维 | "这个场景我没直接参与，但从架构设计角度，我认为..." |
   - 若是平移：你用过什么类似的技术？它们解决的核心痛点是不是一样的？
   - 若是变维：你虽然不懂源码，但你在业务中怎么用它的？遇到过什么坑？
   - 完成标准: 输出一段 150 字左右的转折话术，成功将话题引向用户的主场。
   - **[检查点]**: 暂停等待用户补充其"替代经验"。

4. **总结输出**
   - 将坦诚和转折合并，生成一段完整的高情商应答模板。
   - **完整应答示例**：
     ```
     【平移示例】"Kafka的源码我确实没有深入读过，不过我在上家公司负责过基于RabbitMQ的消息系统，对消息队列的核心设计理念——比如分区、副本、消费模型——都有实战经验。如果让我设计一个类似的系统，我会从这几个角度考虑..."
     
     【降维示例】"Elasticsearch的底层Lucene源码我没有研究过，但在实际项目中，我负责过ES集群的调优和索引设计，处理过千万级数据的检索性能问题。我可以分享一下当时遇到的坑和解决方案..."
     ```
   - 完成标准: 产出最终可供用户背诵和实战的 STAR 结构回答段落。

### B — 边界 (Boundary) ★

#### 不要在以下情况使用此 skill

- 如果面试官考的是极其基础的常识（如"什么是 HTTP"），你连这也想平移，那只会显得基本功太差，本策略救不了硬伤太多的情况。
- 面试官要求你在白板上写出具体的算法代码（如快排）。此时不存在"我不会快排，但我能给你讲讲它在业务里的应用"，写不出就是写不出。

#### 作者在书中警告的失败模式

- **不懂装懂的诚信红线**：试图用假大空的词汇去糊弄面试官，一旦被追问细节就会全盘崩溃，直接触及诚信底线。

#### 容易混淆的邻近方法论

- 顾左右而言他（这是心虚的逃避，而降维平移是坦诚后的主动展示，两者气场完全不同）。

### R — 原文 (Reading)

## 场景 (Scenario)

### A1 — 书中的应用 (Past Application)

#### 案例 1: 面对不熟悉的底层机制
- **问题**: 被面试官问到某个框架非常深的底层原理，而自己平时只做业务调用，确实不懂。
- **方法论的使用**: 采用"变化维度"策略。
- **结论**: 坦言未曾深究源码（内部维度），但立刻切换到"外部应用维度"。
- **结果**: 向面试官详细讲解了该框架在复杂业务线上的实际选型对比、踩坑经验和压测数据，成功赢得了面试官对其实战能力的认可。

### A2 — 触发场景 (Future Trigger) ★

#### 用户会在什么情境下需要这个 skill?

1. 用户在准备面试时，看到某大厂面经里有一个非常偏门或自己未曾涉猎的题目。
2. 用户在模拟面试中卡壳，不知道该怎么把话题圆回来。
3. 用户抱怨自己的项目都是"调包"，一旦被问底层就抓瞎。

#### 语言信号 (用户的话里出现这些就应激活)

- "这个问题我完全没思路，怎么回答比较好？"
- "如果面试官问我没用过的技术栈怎么办？"
- "我只会用 API，不懂源码，被问到怎么说？"
- "面试时遇到盲区怎么圆场？"

#### 与相邻 skill 的区分

- 与 `competency-model-alignment` 的区别: 本技能是专门针对**"不会答的硬伤/盲区"**的战术补救工具，后者是构建**整体回答框架**的战略工具。
