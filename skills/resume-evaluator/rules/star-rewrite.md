# STAR 重写规范（执行细则）

> 本文件是 `resume-evaluator` skill 对低分 bullet 做 STAR 改写的执行规范。AI 必须按本模板重写，不得改变事实。

## 1. STAR 四要素定义

| 要素 | 英文 | 含义 | 简历中的体现 |
|------|------|------|-------------|
| S | Situation | 事情发生的背景/规模/约束 | 一句话交代业务规模或痛点（可省略，融入 T） |
| T | Task | 你的具体职责与目标 | "我负责/我的目标是..." |
| A | Action | 你具体怎么做的、用了什么技术/方法 | 动作动词 + 工具/技术 + 具体手段 |
| R | Result | 可量化的结果 | 指标变化 + 对比基线 |

STAR 要求 Action 最具体（占 bullet 篇幅 50%+），Result 必须量化。Situation 在简历 bullet 中通常压缩为半句背景。

## 2. 简历 bullet 改写模板

### 2.1 标准结构
```
[背景/规模，可选] + 动作动词 + 任务对象 + （用/通过/基于 工具或技能） + 具体手段 + ，使/将/实现 + 指标 + 从/至/提升/降低 + 数值
```

### 2.2 改写四步法
1. **抽动作**：把"负责/参与"换成强动作动词（见 §3 动词表）。
2. **补工具**：在动作后补"用 XX 技术/工具"。
3. **加对象**：明确动作作用的对象（系统/模块/流程）。
4. **嵌指标**：结尾补"使 XX 指标从 A 到 B"。

### 2.3 模板示例
- `主导订单系统重构，基于 Redis 缓存与消息队列削峰，将核心接口 P99 从 800ms 降至 120ms，支撑双十一 50W QPS。`
- `设计监控告警平台，基于 Prometheus + Grafana 采集 200+ 服务指标，将故障平均发现时间从 15 分钟缩短至 2 分钟。`

## 3. 动作动词分级表

| 层级 | 动词 | 适用场景 |
|------|------|---------|
| 强（主导/创造） | 主导、设计、架构、从 0 到 1 搭建、重构、发起 | 核心贡献、技术决策 |
| 中（执行/优化） | 优化、开发、实现、排查、调优、重构、改造 | 日常开发有亮点 |
| 弱（参与/协助） | 参与、协助、负责、跟进、配合 | 应避免或仅在配角场景使用 |

> 规则：bullet 开头动词必须是"强"或"中"层级。"负责/参与"开头应改写或删除。

## 4. 量化指标参考表（按工作类型）

| 工作类型 | 可量化指标 |
|---------|-----------|
| 后端开发 | QPS/TPS、P99/P95 耗时、接口成功率、错误率、吞吐量 |
| 前端开发 | 首屏加载时间、FCP/LCP、包体积、转化率、白屏率 |
| 算法/数据 | 模型准确率/AUC、召回率、训练耗时、A/B 实验提升、GMV 提升 |
| 测试/QA | Bug 发现数、缺陷逃逸率、自动化覆盖率、回归耗时 |
| 运维/SRE | 可用性（99.9x%）、故障 MTTR、告警准确率、成本节省 |
| 带人/管理 | 团队规模、新人上手周期、团队产出、离职率、晋升人数 |
| 修 Bug | Bug 修复数、线上故障率下降、投诉减少、复现率 |
| 写报表/数据 | 报表生成耗时、数据准确率、决策采纳率、节省人力 |
| 代码重构 | 代码行数减少、圈复杂度下降、性能提升、可维护性评分 |
| 客户/业务 | 服务客户数、NPS、续约率、客诉下降、响应时长 |

> 规则：若用户实在无法提供数字，走 fallback——引用"同事/老板对此事的正面评价"或"避免的损失"，但需明确标注"定性证据"且 Result 段降级处理。

## 5. 失败模式（必须避免）

### 5.1 流水账模式
- 表现：按时间顺序罗列"做了 A、做了 B、做了 C"，无重点无结果。
- 诊断：每个 bullet 都以"负责/参与"开头，缺 Result。
- 修复：合并同类项，按"价值贡献"而非"时间顺序"组织，每条补 Result。

### 5.2 无量化模式
- 表现：全是定性描述"提升了效率、优化了体验"。
- 诊断：bullet 无任何数字。
- 修复：按 §4 指标表追问具体数字，无数字则引用定性证据并标注。

### 5.3 团队泛泛而谈模式
- 表现："我们团队完成了 XX"，无法区分个人贡献。
- 诊断：主语全是"我们/团队"，无"我"。
- 修复：明确"我"的角色——"我主导 X 部分，团队整体完成 Y"。

### 5.4 职责描述伪装成成果
- 表现："负责订单系统的开发与维护"（这是职责不是成果）。
- 诊断：bullet 描述的是"岗位说明书"而非"做了什么"。
- 修复：改为"做了什么 + 怎么做 + 结果"。

## 6. 改写前后对比示例

### 示例 1：后端开发
- **改写前**：`参与订单系统开发，负责后端接口。`
  - 诊断：弱动词"参与"，无工具无结果。
- **改写后**：`主导订单核心接口重构，基于 Redis 缓存热点数据并引入消息队列异步削峰，将下单接口 P99 耗时从 800ms 降至 120ms，支撑双十一 50W QPS 零故障。`
  - 改写点：参与→主导；补工具 Redis/MQ；补指标 P99 800→120ms + 50W QPS。

### 示例 2：测试工程师
- **改写前**：`负责测试工作，发现了很多 bug。`
  - 诊断："很多"不可衡量，无工具无场景。
- **改写后**：`搭建接口自动化测试框架（Pytest + Allure），覆盖 300+ 核心接口，将回归测试耗时从 2 天缩短至 4 小时，线上缺陷逃逸率下降 60%。`
  - 改写点：补工具 Pytest/Allure；"很多"→300+；补耗时 2 天→4 小时 + 缺陷率 -60%。

### 示例 3：运维
- **改写前**：`维护线上系统稳定。`
  - 诊断：纯职责，无动作无结果。
- **改写后**：`设计全链路监控告警体系（Prometheus + Grafana + Alertmanager），覆盖 200+ 微服务节点，将故障平均发现时间从 15 分钟缩短至 2 分钟，年度可用性达 99.95%。`
  - 改写点：职责→动作"设计"；补工具栈；补 MTTR 15→2 分钟 + 可用性 99.95%。

### 示例 4：带新人（非技术岗也可量化）
- **改写前**：`带新人，帮助团队成长。`
  - 诊断：无量化，"帮助成长"空泛。
- **改写后**：`制定新人 onboarding 文档与结对机制，使 3 名新人独立产出周期从 6 周缩短至 2 周，其中 1 人在转正考核获评优秀。`
  - 改写点：补机制"文档+结对"；补周期 6→2 周 + 量化人数 3 名。

## 7. 重写输出格式

每个重写建议输出三段：
```
【原句】{原 bullet}
【问题】{诊断出违反的失败模式 + 缺失要素}
【重写】{STAR 重写后的 bullet}
```

> 规则：重写不得改变事实（不能编造数字、不能虚构工具），只能重组结构与补充追问得到的真实信息。若原 bullet 信息不足以重写完整 STAR，输出"需向用户追问：{具体问题}"而非编造。

## 8. 与代码实现的对齐

本规范与 `src/features/resume/star-rewriter.ts` 实现对齐：
- 代码输出 4 段流式内容：我的分析 / STAR 改写 / 底层心法 / 建议。
- 本规范聚焦"STAR 改写"段的内容质量，其余段由 `competency-alignment` 与 `blind-spot-navigation` 供给。
- `src/features/resume/diagnostics/impact.ts` 已实现"职责描述缺量化"的自动检测（`looksLikeResponsibilityOnly`），本规范是其修复执行层。
