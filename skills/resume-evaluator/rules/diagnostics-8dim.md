# 8 维诊断标准（执行细则）

> 本文件是 `resume-evaluator` skill 的诊断核心细则。AI 必须严格按本标准逐维打分（0-10 分制），不得凭主观印象给分。

## 通用评分规则

- 每维独立打分，取值 0-10 的整数或一位小数。
- 评分必须基于客观证据（bullet 原文、时间区间、技能词出现位置），证据缺失时不得给高分。
- 每维打分须附带 1-3 条"扣分依据"（引用简历原文片段）。
- 0-10 分制通用分档：
  - **0-2**：严重缺陷，几乎不可用（如模块完全缺失、全是流水账无任何量化）。
  - **3-4**：明显短板，需较大改写。
  - **5-6**：基本合格，但有提升空间。
  - **7-8**：良好，符合岗位预期。
  - **9-10**：优秀，可作为同类岗位的范本。

## 维度 1：结构完整性 (Structural Completeness)

### 定义
简历必备模块是否齐备、层次是否清晰、信息是否易于 ATS 与 HR 定位。

### 评分标准
| 分数 | 标准 |
|------|------|
| 0-2 | 缺失 3 个以上必备模块；或模块顺序混乱无法阅读 |
| 3-4 | 缺失 1-2 个必备模块；或模块存在但层级混乱 |
| 5-6 | 5 个必备模块齐备；但某模块信息稀疏（如技能栏仅 2 项） |
| 7-8 | 5 个模块齐备且层次清晰；教育/技能/工作分配合理 |
| 9-10 | 模块齐备 + 含个人亮点总结 + 项目与工作经历分离且各有 bullet |

必备模块（缺一即扣分）：基本信息（姓名/联系方式/求职意向）、工作经历、项目经历、技能、教育背景。

### 检查方法
1. 逐项核对 5 个必备模块是否存在。
2. 检查模块顺序是否符合"基本信息 → 总结（可选）→ 工作经历 → 项目经历 → 技能 → 教育"惯例。
3. 检查每个模块是否有明确标题分隔。
4. 检查基本信息是否含至少 2 种联系方式。

### 常见问题模式
- 缺"项目经历"模块，工作与项目混在一起。
- 缺"求职意向"，HR 无法判断目标岗位。
- 技能栏堆砌但无分类（如不分"编程语言/框架/工具"）。
- 教育背景放在最前导致工作经历被折叠。

### 修复建议模板
- `建议在【{模块名}】位置补充【{缺失内容}】，HR 在 6 秒扫读时优先寻找该模块。`
- `建议将【{模块A}】移至【{模块B}】之前，符合 {岗位级别} 候选人的阅读惯例。`

## 维度 2：内容量化度 (Quantification Degree)

### 定义
工作/项目 bullet 中含可衡量数字指标的比例与质量。量化是简历可信度的核心证据。

### 评分标准
| 分数 | 标准 |
|------|------|
| 0-2 | 全部 bullet 无任何数字；纯职责描述 |
| 3-4 | 量化 bullet 占比 < 30% |
| 5-6 | 量化 bullet 占比 30%-60%；但多为笼统数字（如"大量用户"） |
| 7-8 | 量化 bullet 占比 60%-85%；含具体业务指标（QPS/DAU/GMV/耗时） |
| 9-10 | 量化 bullet 占比 > 85%；每个 bullet 含"动作+对象+指标+对比基线" |

### 检查方法
1. 统计工作经历 + 项目经历的 bullet 总数 N。
2. 统计含数字（含 %、万、倍、QPS、DAU、ms、s 等单位）的 bullet 数 M。
3. 计算量化率 = M / N。
4. 检查数字是否含对比基线（如"从 X 降至 Y"，仅有 Y 不含基线降一档）。

### 常见问题模式
- "负责 XX 系统开发"（纯职责，无结果）。
- "服务了大量用户"（"大量"不可衡量）。
- "性能提升 30%"（无基线，不知从多少提升到多少）。
- 数字与职责脱节（列了数字但不说该数字由什么动作产生）。

### 修复建议模板
- `该 bullet 缺量化：原句「{原 bullet}」。建议补「{动作}使{指标}从{基线}提升/降至{新值}」。`
- `「大量/若干/部分」不可衡量，请替换为具体数字（如 100W+ DAU、50ms P99）。`

## 维度 3：STAR 叙事质量 (STAR Narrative Quality)

### 定义
bullet 是否具备 Situation-Task-Action-Result 完整闭环，而非流水账式职责罗列。

### 评分标准
| 分数 | 标准 |
|------|------|
| 0-2 | 全是"负责/参与/协助"开头的职责罗列，无 Action 与 Result |
| 3-4 | 多数 bullet 缺 Result；Action 仅写"做了什么"不写"怎么做" |
| 5-6 | 部分 bullet 含 Action+Result，但 Situation/Task 缺失（不知背景） |
| 7-8 | 多数 bullet 含完整 STAR；Action 具体到工具/方法 |
| 9-10 | 全部 bullet 完整 STAR；Result 含量化且 Action 展现技术决策 |

STAR 四要素：Situation（背景/规模）、Task（你的职责目标）、Action（具体怎么做的、用了什么）、Result（结果指标）。

### 检查方法
1. 抽取每个 bullet，标注是否含 S/T/A/R 各要素。
2. 检查动词开头：是否用"主导/设计/重构/优化"等强动作动词，而非"参与/协助/负责"等弱动词。
3. 检查 Action 是否具体到工具/技术（如"用 Redis 缓存"而非"做了缓存"）。
4. 检查 Result 是否独立成段或明确标注。

### 常见问题模式
- "参与 XX 项目，负责后端开发"（缺 S/T，A 模糊，无 R）。
- "使用 Java 开发了订单系统"（A 有但无 S 与 R）。
- 团队泛泛而谈："我们团队完成了 XX"（无法体现个人贡献）。
- 弱动词堆砌："负责、参与、协助"贯穿全文。

### 修复建议模板
- `该 bullet 缺 STAR 闭环：原句「{原 bullet}」。按「{背景规模}下，我{动作动词}{技术/方法}{具体任务}，使{指标}{变化}」重写。`
- `「我们团队」无法体现个人贡献，请改为"我主导/我设计/我独立完成"。`

## 维度 4：时间线连续性 (Timeline Continuity)

### 定义
工作/教育经历的时间段是否连续，空窗期是否可解释，是否存在时间矛盾。

### 评分标准
| 分数 | 标准 |
|------|------|
| 0-2 | 多处时间矛盾（如两段经历重叠且非兼职）；或无任何时间标注 |
| 3-4 | 存在 > 6 个月未解释空窗；或时间格式不统一无法比对 |
| 5-6 | 有 3-6 个月空窗但未说明；时间格式基本统一 |
| 7-8 | 时间连续；格式统一；空窗 ≤ 3 个月或已说明 |
| 9-10 | 时间连续 + 每段经历标注起止年月 + 晋升/转岗轨迹清晰 |

### 检查方法
1. 按时间排序所有工作经历，检查是否有 > 3 个月空窗。
2. 检查是否有时间重叠（同期两段全职经历）。
3. 检查时间格式是否统一（YYYY.MM vs YYYY 年 MM 月混用扣分）。
4. 检查最近一段经历是否持续至今或近期结束（停更 > 2 年扣分）。

### 常见问题模式
- 两段经历时间重叠且无"兼职/顾问"说明。
- 跨年度空窗（如 2019-2021 无任何记录）。
- 时间格式混乱："2020.3"、"2020 年 3 月"、"March 2020"混用。
- 教育与工作时间倒置（先工作后全日制读研却无说明）。

### 修复建议模板
- `检测到 {起} 至 {止} 存在 {N} 个月空窗，建议补充说明（如：备考/创业/家庭照护/技能进修）。`
- `【{公司A}】与【{公司B}】时间重叠，若为兼职请注明，否则会被质疑真实性。`

## 维度 5：关键词密度 (Keyword Density)

### 定义
简历中核心技术词、能力词的覆盖广度与出现频次，是否匹配目标岗位的高频词。

### 评分标准
| 分数 | 标准 |
|------|------|
| 0-2 | 技术词 < 5 个；或与目标岗位完全无关 |
| 3-4 | 关键词覆盖目标岗位高频词 < 30% |
| 5-6 | 覆盖 30%-60%；但关键词集中在技能栏未在经历中出现 |
| 7-8 | 覆盖 60%-80%；关键词在工作/项目中均有出现 |
| 9-10 | 覆盖 > 80%；关键词自然嵌入经历上下文而非生硬堆砌 |

> 当有目标 JD 时，本维度评分应与 `rules/ats-scoring.md` 的覆盖率交叉校验：ATS 覆盖率 ≥75% 对应本维 ≥7 分。

### 检查方法
1. 若有 JD：按 `rules/ats-scoring.md` 提取 JD 关键词，计算在简历中的覆盖率。
2. 若无 JD：基于岗位类型（如后端/前端/算法）使用通用高频词表，统计覆盖率。
3. 检查关键词是否仅出现在技能栏（"孤立关键词"扣分），需在经历中有上下文。
4. 检查是否有同义词覆盖（如 JD 写"Kubernetes"而简历只写"K8s"，应都出现）。

### 常见问题模式
- 技能栏列了 30 个词但经历中从未体现（孤立关键词）。
- 只用中文缩写不用英文全称（或反之），ATS 检索不到。
- 关键词堆砌在技能栏，工作描述却用泛化表述（如"用了相关技术"）。
- 缺少 JD 中的核心硬技能词。

### 修复建议模板
- `JD 高频词「{关键词}」在简历中未出现，建议在【{经历/项目}】的 bullet 中自然嵌入。`
- `技能栏列「{技能}」但经历无证据，要么补充相关经历，要么从技能栏移除（见维度 6）。`

## 维度 6：技能-经历一致性 (Skill-Experience Consistency)

### 定义
技能栏声明的技能是否在工作/项目经历中有实际使用证据。防止"技能注水"。

### 评分标准
| 分数 | 标准 |
|------|------|
| 0-2 | 技能栏大量堆砌但经历中 < 20% 有证据支撑 |
| 3-4 | < 50% 的核心技能在经历中有上下文 |
| 5-6 | 50%-70% 的核心技能有经历证据 |
| 7-8 | > 70% 的核心技能在经历中可见使用场景 |
| 9-10 | 几乎所有技能均有经历佐证 + 标注熟练度且熟练度与证据匹配 |

### 检查方法
1. 提取技能栏每个技能词。
2. 在工作/项目 bullet 中检索该技能词是否出现（含同义词）。
3. 计算"有证据技能数 / 总技能数"。
4. 检查熟练度声明（精通/熟悉/了解）是否与证据强度匹配：声明"精通"但仅有 1 次提及 → 降级。

### 常见问题模式
- 技能栏写"精通 Kubernetes"但经历中无任何 K8s 相关描述。
- 声明"精通"但证据仅是"用过"（无架构设计/调优/排障场景）。
- 技能栏列了 5 年前用过但近年未提及的技能（生疏却仍声明熟练）。
- 熟练度虚高：所有技能都标"精通"。

### 修复建议模板
- `技能「{技能}」声明「{熟练度}」但经历仅 {N} 处提及，建议降级为「{实际熟练度}」或补充经历证据。`
- `技能栏 {N} 项中 {M} 项无经历佐证，建议移除无证据项，保留有据可查的技能。`

## 维度 7：错别字/格式 (Typos & Format)

### 定义
拼写、标点、排版、中英文混排规范性。这是 ATS 解析与 HR 第一印象的基础。

### 评分标准
| 分数 | 标准 |
|------|------|
| 0-2 | 错别字 ≥ 5 处；或排版混乱无法解析（如表格/图片化文本） |
| 3-4 | 错别字 3-4 处；或中英文标点混用严重；或 bullet 长度参差极大 |
| 5-6 | 错别字 1-2 处；标点基本规范；格式有少量瑕疵 |
| 7-8 | 无明显错别字；中英文间有空格；bullet 长度均衡 |
| 9-10 | 零错别字 + 中英文排版规范 + bullet 统一以动词开头 + 无冗余装饰符号 |

### 检查方法
1. 逐行扫描错别字（含同音字误用、英文拼写错误）。
2. 检查中英文标点混用（如中文句号用了"."、英文逗号用了"，"）。
3. 检查中英文之间是否有空格（规范要求有）。
4. 检查 bullet 是否统一以动词开头、长度是否均衡（建议 15-40 字）。
5. 检查是否有图片化文本/复杂表格（ATS 无法解析）。

### 常见问题模式
- "的的"、"了了"等重复字。
- 英文拼写：Kubernates（应为 Kubernetes）、Recieve（应为 Receive）。
- 中文用英文句号"."结尾。
- 中英文无空格："使用Java开发" 应为"使用 Java 开发"。
- bullet 长度两极分化（一段 5 字一段 100 字）。
- 使用特殊符号（如 ✓ ★ ●）ATS 可能无法解析。

### 修复建议模板
- `检测到错别字：「{原词}」→ 应为「{正确词}」（位置：{location}）。`
- `中英文之间建议加空格：「{原文}」→「{改后}」，提升可读性。`
- `该 bullet {N} 字过长，建议拆为 2 条聚焦不同结果的 bullet。`

## 维度 8：差异化亮点 (Differentiated Highlights)

### 定义
简历是否有区别于同级别候选人的稀缺价值点，而非"流水线工程师"式模板。

### 评分标准
| 分数 | 标准 |
|------|------|
| 0-2 | 全是 CRUD/修 bug 流水账，无任何独特价值 |
| 3-4 | 有少量非日常工作，但未提炼出亮点 |
| 5-6 | 有 1-2 个可识别的亮点（如主导某模块/工具沉淀） |
| 7-8 | 有 2-3 个稀缺亮点（如 0-1 搭建/跨团队推动/技术决策） |
| 9-10 | 亮点密集且稀缺（如开源贡献/专利/架构创新/业务从 0 到 N） |

> 本维度得分 < 5 时，触发编排 `highlight-extractor` 子 skill 榨取亮点素材。

### 检查方法
1. 识别"非日常工作"的描述（如主导、0-1、架构设计、跨团队、开源、专利）。
2. 评估亮点的稀缺性：是否多数同级别候选人都能写出来？
3. 检查亮点是否有量化结果支撑（仅"主导"无结果不算强亮点）。
4. 检查是否有个人总结/技术博客/开源项目等加分项。

### 常见问题模式
- 全部 bullet 是"开发了 XX、维护了 XX"，无任何主动性/创新性。
- 亮点未量化："主导了重要项目"（重要程度未知）。
- 亮点与岗位无关（如应聘后端但亮点全是前端项目）。
- 把团队成果当个人亮点（"团队获得了 XX 奖"未说明个人贡献）。

### 修复建议模板
- `未检测到差异化亮点，建议调用 highlight-extractor 从以下经历榨取：{候选经历列表}。`
- `该亮点缺量化：原句「{原 bullet}」。建议补充「{你的具体角色} + {稀缺动作} + {结果指标}」。`

## 跨维度一致性校验

完成 8 维打分后，执行以下交叉校验：
- **维度 5 与维度 6 互校**：关键词密度高但技能-经历一致性低 → 关键词为"孤立堆砌"，两维均不应给 7+。
- **维度 2 与维度 3 互校**：量化度高但 STAR 质量低 → 可能是"硬塞数字"无叙事逻辑，维度 3 应相应下调。
- **维度 7 否决项**：若错别字 ≥ 5 处，整体诊断报告置信度降一档（HR 会认为态度不严谨）。
