# 能力评估三重境界 (基础/熟练/精通)

## R — 原文 (Reading)

> 基础意味着"会做"：意思是能够独立完成任务，而不是别人想好之后告诉你。
> 熟练意味着"做好"：掌握了做事的最佳实践，能够保证效率和质量。
> 精通意味着"优化"：能够优化当前级别的事情，采取不同的方式、思维和工具来做同样的事，并取得突破。做好只是掌握别人总结的经验，优化意味着你自己创造了新的经验。
>
> — 李运华, 《大厂晋升指南》第4章

---

## I — 方法论骨架 (Interpretation)

在职场评价体系中，技能的掌握程度不能靠感觉，而有一套明确的行为标尺：
1. **基础**：摆脱依赖，能独立把事情做出来。
2. **熟练**：运用行业或团队的"最佳实践"，高效率、低错误率地交付标准结果。
3. **精通**：跳出既有框架，发现现有流程或系统的问题，并成功引入新工具、新架构或新思维，为团队"创造了新经验"。
这三重境界是阶梯式的，只有达到"精通"，才算彻底吃透了当前岗位，具备了向上突破的基础。

---

## A1 — 书中的应用 (Past Application)

### 案例 1: 阶梯式职级（如腾讯T2系）的能力要求划分
- **问题**: 在腾讯旧版职级中，T2.1, T2.2, T2.3 做的事情看起来差不多，评委很难客观区分。
- **方法论的使用**: 用三重境界对标。T2.1 是基础（能写代码），T2.2 是熟练（能高质量写出符合规范的代码），T2.3 是精通（能重构代码或引入新框架优化开发效率）。
- **结论**: 晋升不同的小等级时，不能只拿"我做过什么"说事，必须拿出符合该等级行为特征的证据。
- **结果**: 使得员工在准备答辩时，明确知道要提炼出"我如何优化了团队开发模式"的证据来申请精通级（如T2.3）的晋升。

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. 写简历或述职PPT时，不知道怎么写才能显得自己"牛"，只能堆砌"参与了XX项目"。
2. 纠结于"我是不是已经掌握了 Vue/React/Java"，不知道怎么衡量自己的技术水平。
3. 想知道自己手头重复性的工作还能怎么做出花样来。

### 语言信号 (用户的话里出现这些就应激活)

- "我的简历这部分怎么优化一下？"
- "我答辩PPT里写了熟练掌握Java，这样写好吗？"
- "这个业务我都做了两年了，闭着眼睛都能写，还能怎么提升？"
- "怎么证明我比同事更强？"

### 与相邻 skill 的区分

- 与 `jinsheng-dicing-luoji` 的区别: 底层逻辑教你"去拿下一级别的任务"，而本 skill 教你"怎么把当前任务从熟练拔高到精通"。

## E — 可执行步骤 (Execution)

当 skill 被激活后, agent 应按以下步骤执行:

1. **确认评估对象足够具体**
   - 先要求用户只选定 1 项具体技能、业务模块或工作职责，不接受"我整体很强/很弱"这种泛化表述。
   - **[检查点]**: 必须暂停，等待用户给出具体评估对象。
   - 判停条件：如果用户给不出具体对象，终止细分判断，先帮助其缩小到单一技能或单一职责。
   - 完成标准: 将评估范围收敛到一个可被举证的对象。

2. **识别当前状态**
   - 要求用户描述一项他自认为"做得很好"的日常工作或技能。
   - **三重境界判断标准**：
     | 境界 | 行为特征 | 简历/答辩表述 |
     |------|---------|--------------|
     | 基础 | 能独立完成，不依赖他人指导 | "能够独立完成XX功能开发" |
     | 熟练 | 运用最佳实践，高质高效交付 | "熟练掌握XX，代码质量符合团队规范" |
     | 精通 | 优化流程/工具，为团队创造新经验 | "主导XX优化，效率提升N%，沉淀为团队最佳实践" |
   - **[检查点]**: 暂停等待用户回答。
   - 完成标准: 根据回答，准确定位用户在该技能上的真实层级，并输出诊断结果。

3. **暴露"熟练"的假象**
   - 判停条件：如果用户已处于"精通"或"基础"，跳到步骤 4。
   - 如果用户处于"熟练"，向他指出：熟练只是代表你是一个合格的执行者，没有创造增量价值，单纯的"快和不出错"不够晋升。
   - 完成标准: 明确向用户解释熟练与精通的本质区别。

4. **制定"精通"突破方案**
   - 引导用户思考并提问：在这个任务的上下游，有什么痛点？能否引入业界的新框架？能否写个脚本自动化？能否抽象成通用组件给别的团队用？
   - **精通突破方向清单**：
     ```
     【工具优化】能否写脚本/工具提升效率？如：自动化测试、代码生成器
     【流程优化】能否简化/标准化流程？如：代码评审checklist、上线流程
     【知识沉淀】能否输出文档/培训？如：最佳实践文档、新人培训体系
     【架构优化】能否重构/抽象？如：公共组件、SDK封装
     ```
   - **[检查点]**: 暂停等待用户回答；若用户暂时给不出方向，再提供 2-3 个突破建议。
   - 完成标准: 输出 1-2 个能产生"创造新经验 / 优化当前事情"的具体行动点。

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 用户在学习一个全新的、完全不熟悉的领域时（此时应先追求"基础"，谈精通为时尚早）。

### 作者在书中警告的失败模式

- 把"做得快、出错少、很省心"误判成"精通"。这最多证明你已经熟练，仍然没有说明你是否优化了流程、沉淀了方法，或为团队创造了新经验。
- 在答辩或简历里只写"参与过什么项目"，却拿不出"如何优化现有做法"的证据。结果看起来经历很多，但无法支撑更高一级的能力判断。

### 作者的盲点 / 时代局限

- "创造新经验"在很多成熟度极高、管控极严的大型系统里是不被允许的（即"别乱动祖传代码"）。此时追求"精通"可能会带来政治风险或生产事故，需要结合团队文化判断。

### 容易混淆的邻近方法论

- 容易和单纯的"技术深度挖掘"（如看底层源码）混淆。看源码如果不产生实际的流程/代码优化，依然不算这里的"精通"。
- 容易和 `jinsheng-dicing-luoji` 混淆。底层逻辑讨论的是"如何越级打怪"，本 skill 讨论的是"你对当前工作到底只会做、能做好，还是能优化"。
- 容易和 `jinsheng-san-yuanze` 混淆。三原则是任务筛选器，本 skill 是能力分层尺子。
