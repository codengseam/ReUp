# 晋升三大核心原则 (主动/成长/价值)

## R — 原文 (Reading)

> 1. 主动原则：主动规划工作任务，主动跟别人了解更多信息。主动做事的人，比等着别人安排的人更容易晋升。
> 2. 成长原则：一边做事一边挖掘成长点、提升自己能力的人，比光顾着做事的人更容易晋升。
> 3. 价值原则：学习能为公司产出价值的技能。在投入时间和精力学一项技能之前，先思考一下能为公司带来什么。
>
> — 李运华, 《大厂晋升指南》第3章

---

## I — 方法论骨架 (Interpretation)

在职场中，不是所有努力都能转化为晋升资本。这三条原则是检验你每天做的事是否"有用"的试金石：
- **主动**：不只是"听话照做"，而是主动向上沟通意愿，跨部门获取业务反馈（如上线后效果如何）。
- **成长**：不做低水平重复劳动（如把1年的经验重复10年），要在每次任务中挖掘"能让我能力质变"的挑战点，做复盘总结。
- **价值**：这是最残酷也最现实的一条——你的能力提升如果不能直接变现为公司的业务价值，在晋升评委眼里就是无效的。放弃不能立刻落地的"屠龙之技"（如普通前端强啃Linux内核）。

## A1 — 书中的应用 (Past Application)

### 案例 1: 业务开发去学"编译原理"的陷阱
- **问题**: 一个做 Android App 业务开发的同学，花了半年时间死磕"编译原理"，觉得这是编程语言的基础，学会了肯定有好处。
- **方法论的使用**: 用价值原则审视。学编译原理对当前的 App 业务开发有什么直接帮助吗？能开发出新功能还是能显著提升性能？
- **结论**: 几乎没有。这段学习时间对"公司价值"产出为零，在晋升中不会得到任何加分。
- **结果**: 该同学及时止损，转向学习与当前业务更紧密相关的技术（如 Android 性能优化、架构设计），将能力转化为业务成果。

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. 面对新技术层出不穷，不知道该学哪个，感觉什么都得学时。
2. 觉得自己每天像"拉磨的驴"，很忙但很空虚，不知道出路在哪时。
3. 和主管沟通定目标（OKR）前，不知道怎么提要求时。

### 语言信号 (用户的话里出现这些就应激活)

- "我最近想学 Rust / Go / 某某新技术，你觉得有用吗？"
- "我每天就是接需求、写CRUD代码，怎么破局？"
- "怎么和老板谈我想做更有挑战的事情？"
- "我该不该去接手那个大家都不愿意做的模块？"

### 与相邻 skill 的区分

## E — 可执行步骤 (Execution)

当 skill 被激活后, agent 应按以下步骤执行:

1. **确认目标是否真的是"为了晋升"**
   - 先问用户：你现在是在做晋升取舍、工作规划，还是只是出于兴趣随便学学/想轻松一点？
   - **[检查点]**: 必须暂停，等待用户确认目标后再继续。
   - 判停条件：如果用户明确表示目标不是晋升或公司价值，而是兴趣体验、休息放松、工作生活平衡，则终止本 skill，不要用"价值原则"强压用户。
   - 完成标准: 明确这次讨论是否适合用"主动/成长/价值"做决策过滤。

2. **价值过滤 (Value)**
   - 让用户列出他想做的事或想学的技术。
   - **价值评估清单**：
     | 学习/任务选项 | 价值判断问题 | 判定标准 |
     |--------------|-------------|---------|
     | 新技术学习 | 能帮公司多赚钱/省钱/提效/降风险吗？ | 能→保留，不能→砍掉 |
     | 新项目接手 | 能解决团队痛点或业务瓶颈吗？ | 能→保留，不能→砍掉 |
     | 流程优化 | 能提升团队整体效率吗？ | 能→保留，不能→砍掉 |
   - **[检查点]**: 暂停等待用户回答。
   - 完成标准: 根据回答，砍掉不能带来直接业务价值的选项（即使它听起来很极客）。

3. **成长诊断 (Growth)**
   - 判停条件：如果没有保留下任何选项，直接终止并告知用户当前选项均对晋升无益。
   - 对剩下的选项提问用户：做这件事，能让你产生"质变"吗？还是只是让你变得更熟练？
   - **成长挖掘话术**：
     ```
     【从熟练到质变】"原本只是修Bug，能否顺便重构该模块？"
     【从执行到设计】"原本只是写代码，能否参与架构设计讨论？"
     【从单点到全局】"原本只负责一个模块，能否了解上下游全链路？"
     ```
   - **[检查点]**: 暂停等待用户回答。
   - 引导用户在常规任务中增加难度（例如：原本只是修Bug，能否顺便重构该模块？）。
   - 完成标准: 为每个保留的任务挖掘出至少 1 个"成长点"。

4. **主动出击 (Initiative)**
   - 询问用户：要完成这些成长点，你需要找谁沟通？（如主管、产品经理、兄弟团队等）
   - **主动沟通话术模板**：
     ```
     【向主管争取】"我对XX方向很感兴趣，近期是否有相关项目可以参与？"
     【跨部门协作】"我注意到XX团队有个痛点，我们这边可以提供支持，是否可以一起推进？"
     【向上反馈】"做完这个项目后，我想了解一下上线后的效果和用户反馈，方便后续优化。"
     ```
   - **[检查点]**: 暂停等待用户回答。
   - 鼓励用户不要等任务派下来，而是主动去拉通资源。
   - 完成标准: 输出一个结构化的主动沟通话术或行动计划（如：下周和老板1v1聊想接手架构优化的具体话术）。

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 用户的核心诉求是"我想摸鱼/躺平"或"寻找工作生活平衡"（晋升三原则极度消耗精力，不适合躺平党）。
- 业余爱好驱动的学习（如果用户学某个语言纯粹是为了好玩，不为了晋升，就不必用"公司价值"来约束他）。

### 作者在书中警告的失败模式

- 误把"听话出活快"等同于"主动和成长"；误把"自己觉得牛逼的技术"等同于"有价值的技术"。

### 作者的盲点 / 时代局限

- 书中默认团队目标、个人成长与公司价值能够较顺畅对齐。但在强依赖、弱授权或目标极度模糊的组织里，个人即使很主动，也可能拿不到高价值任务，此时需要先解决资源和授权问题。

### 容易混淆的邻近方法论

- 容易和"时间管理"混淆。三大原则不是教你如何把时间排满，而是教你如何砍掉"伪努力"。
- 容易和 `jinsheng-dicing-luoji` 混淆。底层逻辑关注的是职级跨越路径，本 skill 关注的是你每天该接什么活、学什么东西。
- 容易和 `nengli-sanzhong-jingjie` 混淆。能力三境界用于判断"你做到了哪一层"，三原则用于判断"这件事值不值得做"。
