# dimension-weights.md — 维度权重表

> 被 `../SKILL.md` 第 6 步与 `matching-model.md` §3 调用。本文件定义按 JD 类型切换的权重配置与动态调整规则，AI 必须先由 `jd-decoder` 判定岗位类型，再查表取权重。

## 1. 权重基础表（按岗位类型）

> 岗位类型由 `jd-decoder` 输出的 `job_type` 字段决定，取值 `tech` / `management` / `general`。无法判定时默认 `general`。

### 1.1 技术岗（tech）

| 维度 | 权重 | 说明 |
|------|------|------|
| 技能（Skill） | 50% | 技术栈深度与广度，必备技能权重更高 |
| 经验（Experience） | 25% | 项目复杂度、量化产出、领域相关性 |
| 学历（Education） | 10% | 技术岗对学历宽容度高，仅作刚性闸门与少量加权 |
| 软技能（Soft Skill） | 15% | 协作、抗压、技术沟通 |
| **合计** | **100%** | |

### 1.2 管理岗（management）

| 维度 | 权重 | 说明 |
|------|------|------|
| 管理经验（含于 Experience 子项） | 35% | 单独拆出管理经验权重，团队规模、汇报层级、跨部门协同 |
| 技能（Skill） | 25% | 业务理解 + 专业硬技能 |
| 经验（业务经验，非管理） | 25% | 行业沉淀、项目操盘 |
| 学历（Education） | 5% | 管理岗对学历要求进一步降低，MBA 等仅作加分 |
| 软技能（Soft Skill） | 10% | 领导力、决策、向上管理 |
| **合计** | **100%** | |

> 说明：管理岗把"经验"拆为"管理经验 35% + 业务经验 25%"，本质仍是 Experience 维度的子拆分，对外仍输出 skill/experience/education/soft_skill 四维得分，experience 维度的内部合成公式见 §3。

### 1.3 通用岗（general）

| 维度 | 权重 | 说明 |
|------|------|------|
| 技能（Skill） | 35% | 通用职业技能（如运营、文案、分析工具） |
| 经验（Experience） | 30% | 岗位相关经验积累 |
| 学历（Education） | 20% | 通用岗学历筛选权重较高（HR 初筛卡学历常见） |
| 软技能（Soft Skill） | 15% | 沟通、执行、学习力 |
| **合计** | **100%** | |

## 2. 动态调整规则

基础权重之上，按 JD 文本信号做"±10% 单维度上浮"调整，调整后必须重新归一化到合计 100%。

### 2.1 关键词密度触发

对 JD 全文做关键词密度统计（关键词出现次数 / JD 总词数 × 1000），密度超过阈值的维度上浮 10%（上限 +10%，不叠加同维度多次触发）：

| 维度 | 触发关键词示例 | 密度阈值（‰） | 调整 |
|------|---------------|--------------|------|
| 技能 | 具体技术名词（React / K8s / SQL / Figma 等可枚举技能词） | ≥ 8‰ | 技能 +10% |
| 经验 | "经验/年限/负责/主导/操盘/从 0 到 1" | ≥ 6‰ | 经验 +10% |
| 学历 | "本科/硕士/985/211/统招/学历" | ≥ 4‰ | 学历 +10% |
| 软技能 | "沟通/抗压/协作/领导/驱动/跨部门" | ≥ 5‰ | 软技能 +10% |

### 2.2 归一化算法

设基础权重向量为 `W_base`，触发上浮的维度集合为 `U`（每个上浮 +10 个百分点），则：

```
W_adjusted[i] = W_base[i] + (10 if i in U else 0)
W_final[i] = W_adjusted[i] / Σ W_adjusted      # 归一化回 100%
```

### 2.3 调整示例

技术岗基础权重（50/25/10/15），若 JD 技能词密度 9‰（触发）且经验词密度 7‰（触发）：

```
W_adjusted = (60, 35, 10, 15)   # 技能+10，经验+10
Σ = 120
W_final = (50.0%, 29.2%, 8.3%, 12.5%)
```

### 2.4 硬规则

- 单维度最多上浮一次 +10%，不累加。
- 调整后若某维度权重 < 3%，该维度仍保留参与计算（刚性层学历除外）。
- 动态调整必须在报告中明示：`weight_adjustments: [{dimension, reason, delta}]`，保证可溯源。

## 3. 管理岗 Experience 子维度合成

管理岗的 experience 维度得分由"管理经验"与"业务经验"两部分合成：

```
experience_score = 0.58 × management_exp_score + 0.42 × business_exp_score
```

- `management_exp_score`：按团队规模、管理层级、管理年限评分（0-100）。无管理经验 → 0；带 3 人以内 → 40；带 5-10 人 → 65；带 10+ 人或跨部门 → 85+。
- `business_exp_score`：按业务领域相关性与项目复杂度评分，沿用通用经验评分通道。

合成比例 0.58/0.42 对应管理岗表中"管理经验 35% / 业务经验 25%"的内部相对比（35:25 ≈ 0.58:0.42）。

---

## 4. 权重不可调项

- 刚性层（学历/年限/证书的布尔闸门）不参与本权重表，权重表只作用于弹性层 0-100 打分后的聚合。
- 任何人工"感觉重要"的维度不得绕过本表自行加权；如需新增维度，须先扩展本文件。
