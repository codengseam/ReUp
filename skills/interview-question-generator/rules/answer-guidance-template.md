# answer-guidance-template.md — 答案与引导模板

> 本文件是 `../SKILL.md` 步骤 6 的路由细则，定义每道题的输出结构与 6 类题型的模板变体。
> 题型分类见 `question-taxonomy.md`；难度分级见 `difficulty-grading.md`；边界与自检见 `boundaries.md`；RAG 外挂见 `rag-hook-spec.md`。

## 通用输出结构（所有题型共用骨架）

每道题须输出以下 7 个字段，缺一不可：

| 字段 | 内容要求 |
|------|---------|
| 1. 题目 | 题干全文；附溯源标注 `resume_ref` / `jd_ref` / `source` 之一 |
| 2. 参考答案 | 结构化三段：核心观点（1-2 句结论）→ 展开论述（原理/逻辑链）→ 代码/示例（可运行的伪代码或场景实例；纯概念题可改为流程图描述） |
| 3. 回答引导 | STAR 结构引导（提示候选人按 Situation-Task-Action-Result 组织）+ 关键追问点（2-4 个层层递进的追问）+ 评分要点（优秀/合格/不合格三档判定） |
| 4. 考察点 | 对应 Spencer 冰山模型六层胜任力维度（知识/技能/社会角色/自我形象/特质/动机），标主考察层 1-2 个，依据 `references/interview-refs.md` §1.1 §1.2 |
| 5. 难度等级 | L1-L4，判定标准见 `difficulty-grading.md` |
| 6. 常见错误回答 | 列 2-3 个典型错误（含避坑提示），覆盖"硬编/不懂装懂""背概念不会用""团队成果冒充个人成果"等 |

**STAR 既作候选人回答骨架，也作面试官追问检查表**：候选人回答缺任一要素即可继续追问。依据 `references/interview-refs.md` §1.3。

**关键追问点须层层递进**：复现白海飞三层递进（表层事实→深度细节→感受观点，见 `references/interview-refs.md` §2.1）与北森三层追问（结果→行为→动机，见 §2.3），任一追问点不得停留在表态层。

**考察点须可观测**：胜任力须有可观测行为佐证（McClelland 1973），考察点描述须落到具体行为信号，不接受"沟通能力强"等空泛表述。

## 变体 1：技术深挖题模板

**模板差异**（相对通用骨架）：参考答案的"代码/示例"段必填（技术题须有代码或机制图示）；关键追问点按"原理→源码→踩坑/观点"三层递进；考察点主压技能层，次压潜力层。

### 完整示例

**1. 题目**
> 你简历里提到用 Redis 做订单缓存，能讲讲缓存雪崩是怎么产生的？你当时是怎么预防的？
- 溯源：`resume_ref` = 简历"项目经历-订单系统：使用 Redis 缓存热点商品数据"

**2. 参考答案**
- 核心观点：缓存雪崩指大量 key 在同一时间集体过期或 Redis 整体宕机，请求全部穿透到数据库压垮 DB。预防核心是"打散过期时间 + 多级兜底"。
- 展开论述：
  1. 过期时间打散：给每个 key 的 TTL 加随机偏移（如 `ttl = base + random(0, 300s)`），避免同时失效。
  2. 互斥重建：缓存 miss 时用分布式锁（如 Redis `SET NX`）限单线程回源，其余线程等待或读旧值，防击穿。
  3. 多级兜底：本地缓存（如 Caffeine）做第一层 + Redis 第二层 + DB 兜底；Redis 宕机时本地缓存扛住部分流量。
  4. 熔断降级：DB 压力超阈值时触发熔断，返回降级数据，保 DB 不崩。
- 代码/示例（伪代码）：
  ```java
  // 过期时间打散
  int baseTtl = 3600;
  int ttl = baseTtl + ThreadLocalRandom.current().nextInt(0, 300);
  redis.setex(key, ttl, value);

  // 互斥重建（防击穿）
  if (redis.get(key) == null) {
      if (redis.set(lockKey, "1", "NX", "EX", 10)) { // 拿锁
          try {
              String v = db.query(key);
              redis.setex(key, ttl, v);
              return v;
          } finally { redis.del(lockKey); }
      } else {
          Thread.sleep(50); return redis.get(key); // 等待重读
      }
  }
  ```

**3. 回答引导**
- STAR 结构引导：让候选人先说 Situation（订单系统缓存场景与 QPS 量级）→ Task（你负责解决什么缓存问题）→ Action（具体做了哪几步预防）→ Result（上线后效果/是否真发生过雪崩）。
- 关键追问点（三层递进）：
  1. 表层事实：你当时缓存了哪些 key？TTL 怎么设的？（扫描经验面）
  2. 深度细节：Redis `SET NX` 的过期时间和互斥锁释放顺序，如果持锁线程崩溃锁怎么释放？（探究原理深度）
  3. 感受观点：如果让你重新选型，什么场景下你会不用 Redis 而用本地缓存为主？（考查选型权衡）
- 评分要点：
  - 优秀：能讲清雪崩与击穿区别，给出打散+互斥+多级三层方案，并能讨论锁续期/宕机场景。
  - 合格：能说清雪崩成因与至少一种预防手段（打散或互斥）。
  - 不合格：混淆雪崩与穿透，或只会背概念说不出预防动作。

**4. 考察点**
- 主：技能层（Redis 缓存机制深度）
- 次：潜力层（踩坑复盘与技术选型判断力）

**5. 难度等级**：L2（应用/对比，对应 P6，依据 `difficulty-grading.md`）

**6. 常见错误回答**
- 硬编：简历其实没真正用过 Redis，把博客概念当亲身经历讲，被追问"TTL 设多少"时露馅。避坑：答不上具体参数时坦诚"当时参数是同事定的，我主要负责 X"，不编造。
- 混淆概念：把缓存雪崩说成缓存穿透（穿透是查不存在的 key，雪崩是 key 集体失效）。避坑：先明确区分三类问题再展开。
- 团队成果冒充：把团队整体的缓存方案说成自己一个人的。避坑：主动说清"打散 TTL 是我做的，互斥锁是同事做的，我参与了联调"。

## 变体 2：项目追问题模板

**模板差异**：参考答案按 STAR 组织（而非三段论）；关键追问点**必须含"个人贡献边界"追问**（依据 `question-taxonomy.md` §2）；考察点主压经验真实性与个人贡献度。

### 完整示例

**1. 题目**
> 你简历里写到"主导订单系统性能优化，接口响应从 800ms 降到 200ms"，能展开讲讲你具体做了哪些动作？这个成果里你和同事各负责哪部分？
- 溯源：`resume_ref` = 简历"项目经历-订单系统性能优化：响应时间 800ms→200ms"

**2. 参考答案（STAR 组织）**
- 核心观点：性能优化需先定位瓶颈（DB/网络/计算），再针对性优化，并用量化指标验证。
- 展开论述（STAR）：
  - S/T：订单详情接口 P99 800ms，大促期间超时率 5%，目标是降到 300ms 以内。
  - A：① 用 APM 链路追踪定位瓶颈在 DB 慢查询与多次 RPC 串行；② 给热点查询加联合索引，慢 SQL 从 200ms 降到 10ms；③ 把串行 RPC 改并行（CompletableFuture），网络耗时从 300ms 降到 120ms；④ 加本地缓存减少重复查询。
  - R：P99 降到 200ms，超时率 5%→0.1%。
- 代码/示例：
  ```java
  // 串行 RPC 改并行
  CompletableFuture<Order> orderF = CompletableFuture.supplyAsync(() -> orderService.get(id));
  CompletableFuture<User> userF  = CompletableFuture.supplyAsync(() -> userService.get(uid));
  CompletableFuture<Item> itemF  = CompletableFuture.supplyAsync(() -> itemService.get(itemId));
  CompletableFuture.allOf(orderF, userF, itemF).join();
  // 网络耗时 max(各 RPC) 而非 sum
  ```

**3. 回答引导**
- STAR 结构引导：强制候选人按 S（业务背景与量级）→ T（你承担的角色与目标）→ A（具体动作，须到代码/配置级别）→ R（量化结果与衡量方式）展开。
- 关键追问点（含贡献边界，层层递进）：
  1. S/T：这个项目当时团队几个人？你的角色是 owner 还是参与者？
  2. A：你具体做了哪几项优化？最大的技术障碍是什么？怎么定位的？
  3. R：200ms 是怎么测的？P99 还是 avg？优化前后监控曲线能描述吗？
  4. 贡献边界（必问）：刚才那些优化里，哪几项是你独立做的？同事负责哪部分？你和他们怎么分工的？
  5. 复盘：如果重做，你会先优化哪部分？为什么？
- 评分要点：
  - 优秀：能说出具体瓶颈定位手段、量化前后对比、清晰个人贡献边界，复盘有反思。
  - 合格：能说出主要优化动作与结果，贡献边界基本清楚。
  - 不合格：含糊说"大家一起做的"，说不出自己具体动作，或结果数字经不起追问。

**4. 考察点**
- 主：技能层（性能优化方法论）+ 经验真实性
- 次：潜力层（复盘能力）

**5. 难度等级**：L2（应用/分析，对应 P6）

**6. 常见错误回答**
- 团队成果冒充个人：把"团队优化"说成"我优化"，被贡献边界追问后露馅。避坑：主动划清边界。
- 数字经不起追问：说"降到 200ms"但说不出是 P99 还是 avg、怎么测的。避坑：提前备好测量口径。
- 缺复盘：只讲做了什么，不讲"如果重做会怎么改"。避坑：每段经历备一句反思。

## 变体 3：系统设计题模板

**模板差异**：参考答案的"代码/示例"段改为架构图描述（模块+数据流）；关键追问点必含 CAP/成本权衡；考察点主压架构设计与社会角色（跨团队协作）；难度多为 L3。

### 完整示例

**1. 题目**
> JD 要求"设计支撑千万级日活的消息推送系统"，给定 QPS 峰值 10 万、消息不丢不重、端到端延迟 < 1s。给出架构并说明 CAP 取舍。
- 溯源：`jd_ref` = JD"岗位职责：高并发场景下消息推送系统设计与开发"

**2. 参考答案**
- 核心观点：推送系统核心是"接入层（长连接）+ 消息层（MQ 持久化）+ 路由层（用户→设备映射）+ 存储层（离线消息）"，CAP 上选 AP（可用+分区容错），用最终一致性换低延迟。
- 展开论述：
  1. 接入层：WebSocket 长连接网关，水平扩展，单机维持 10 万连接，按用户分片。
  2. 消息层：Kafka 持久化，按推送通道分 partition，保证消息顺序与不丢；生产端 ack=all，消费端手动提交 offset。
  3. 路由层：Redis 存 `userId → {connId, gatewayIp}` 映射，用户上线/迁移时更新。
  4. 存储层：离线消息存 Redis（7 天）+ MySQL（长期），用户上线后拉取未读。
  5. CAP 取舍：选 AP，网络分区时允许部分用户消息延迟到达（最终一致），优先保推送可用；不丢不重靠 MQ 持久化 + 消费端幂等（去重表/唯一键）。
- 代码/示例（架构图描述）：
  ```
  [Client] <--WS--> [Gateway 集群] --push--> [Kafka]
                                              |
                          [Route(Redis userId→conn)] <--查询-- [Gateway]
                                              |
                          [Offline Store: Redis+MySQL] <--未读拉取-- [Client 上线]
  ```

**3. 回答引导**
- STAR 结构引导：S（业务场景与约束）→ T（你的设计目标与取舍前提）→ A（架构分层与关键决策）→ R（容量估算与极端场景验证）。
- 关键追问点（含权衡，层层递进）：
  1. 容量估算：千万日活、QPS 10 万，你需要多少台 Gateway？Kafka 多少 partition？怎么算的？
  2. CAP 追问：某机房整体宕机时，你的方案消息会丢吗？延迟多久恢复？你怎么取舍的？
  3. 不丢不重：消费端怎么保证幂等？去重表存哪？存储压力怎么扛？
  4. 极限追问（L4，仅 P8+）：如果流量再翻 10 倍，现有架构哪里先崩？你怎么改造？
- 评分要点：
  - 优秀：给出完整分层架构，容量估算有依据，CAP 取舍清晰，能讨论极端场景。
  - 合格：给出主要模块与数据流，能说清不丢不重手段。
  - 不合格：只会画模块图不会算容量，或回避 CAP 取舍说"都要"。

**4. 考察点**
- 主：技能层（架构设计与规模意识）
- 次：社会角色层（跨团队协作，推送系统涉及多端对接）+ 特质层（权衡判断力）

**5. 难度等级**：L3（设计/优化/权衡，对应 P7，依据 `difficulty-grading.md`）

**6. 常见错误回答**
- 回避权衡：说"既要高可用又要强一致"。避坑：CAP 必须明确取舍，并说明取舍代价。
- 不会算容量：画了架构图但说不出要几台机器。避坑：按单机连接数/QPS 上限做除法。
- 忽略幂等：只说"用 MQ 不丢"，没说消费端去重。避坑：不丢靠 MQ ack，不重靠消费端幂等，二者分开讲。

## 变体 4：行为面试题模板

**模板差异**：参考答案用"具体行为事例"而非概念；关键追问点按北森三层（结果→行为→动机）递进，须追问到"具体说了什么/做了什么"（BEI 深度，依据 `references/interview-refs.md` §1.2）；考察点主压社会角色/自我形象/动机等冰山下层。

### 完整示例

**1. 题目**
> 请举一个你和同事在技术方案上产生严重分歧的具体例子，最后是怎么解决的？请讲具体细节，不要讲观点。
- 溯源：`jd_ref` = JD"任职要求：跨部门沟通协作能力"

**2. 参考答案（BEI 行为事例）**
- 核心观点：分歧解决的关键是"对事不对人 + 用数据/实验兜底 + 给对方面子"。回答必须落到具体事件，禁止表态。
- 展开论述（STAR + BEI）：
  - S：去年做支付重构，我主张用 Saga 分布式事务，同事主张用 TCC，双方僵持一周。
  - T：我需要在不伤关系的前提下推进方案落地。
  - A：① 我没有直接否定 TCC，而是拉了一个 30 分钟会议，把两套方案在"一致性强度/开发成本/回滚难度"三维度做对比表；② 提议用真实流量跑一次灰度实验，看实际数据；③ 实验显示 Saga 在我们的弱一致性场景下开发成本低 40%，同事认可；④ 会上我主动说"TCC 在强一致场景更优，我们这个场景 Saga 更合适"，给同事留台阶。
  - R：方案一周内落地，后续协作未受影响，同事还在别的项目主动找我合作。
- 代码/示例：本题为行为题，无代码；用对比表替代：
  ```
  方案 | 一致性 | 开发成本 | 回滚难度
  Saga | 弱一致 | 低       | 易
  TCC  | 强一致 | 高       | 难
  ```

**3. 回答引导**
- STAR 结构引导：强制"请举一个具体例子"开头，禁止表态式回答（"我觉得沟通很重要"一类直接打断）。
- 关键追问点（北森三层：结果→行为→动机）：
  1. 结果层：最后方案落地了吗？同事关系后续怎么样？（确认结果真实）
  2. 行为层：会议上你具体说了哪句话让同事接受？你做的对比表具体哪一列说服了他？（追问到具体行为，BEI 深度）
  3. 动机层：你当时为什么选择做实验而不是直接找 leader 拍板？（探究行为背后的动机与价值观）
- 评分要点：
  - 优秀：具体到"说了什么/做了什么"，行为细节经得起追问，动机层能反思自己。
  - 合格：有具体事件与结果，行为细节基本清楚。
  - 不合格：只讲观点不讲事件，或事件经不起行为层追问（"我们就沟通了一下就好了"）。

**4. 考察点**
- 主：社会角色层（协作与冲突处理）
- 次：自我形象层（自我反思）+ 动机层（追求推进 vs 追求和谐）

**5. 难度等级**：L2（行为题多为 L2，对应 P6+，依据 `difficulty-grading.md` 职级映射）

**6. 常见错误回答**
- 表态式回答："我觉得沟通很重要，大家坐下来谈谈就好了"。避坑：必须给具体事件，事件须有时间地点人物。
- 抢功/贬低同事："我当时是对的，同事水平不行"。避坑：客观描述分歧，不评判对方能力。
- 缺行为细节：经不起"你具体说了什么"追问。避坑：提前备好关键对话原话。

## 变体 5：反向提问题模板

**模板差异**：参考答案改为"反问意图解读"（候选人为什么这么问）；无 STAR 追问（反问不追问候选人），改为"面试官可能的回答与候选人应对"；难度不分级（依据 `question-taxonomy.md` §5）；考察点压职业成熟度与动机。

### 完整示例

**1. 题目**
> "请问团队目前面临的最大技术挑战是什么？如果我有幸加入，您希望我能在哪些方面快速贡献？"
- 溯源：`jd_ref` = JD"岗位职责"中可刺探的团队现状信息点；反问落在三元交集：应聘者关心成长 ∩ 面试官关心能否干活 ∩ 职位交付挑战（由 `reverse-questioning-framework` 产出，本 skill 透传）

**2. 参考答案（反问意图解读）**
- 核心观点：此反问一举三得——展现对职位真实兴趣、刺探团队真实痛点、暗示自己愿意快速贡献。
- 展开论述：
  1. 第一问"最大技术挑战"：刺探团队现状是否健康（是攻坚期还是填坑期），判断入职后是成长还是背锅。
  2. 第二问"希望我快速贡献"：展现主动性与结果导向，同时把面试官的期望前置化，避免入职后预期错位。
  3. 三元交集落点：应聘者关心成长（挑战=成长机会）∩ 面试官关心能否干活（快速贡献）∩ 职位交付挑战（JD 岗位职责）。
- 代码/示例：本题为反问题，无代码；给出"面试官可能回答 + 候选人应对"：
  ```
  面试官可能答："我们最大的挑战是把吞吐从 1 万 QPS 提到 5 万，架构要重构。"
  候选人应对：① 接住并关联自己经历："我之前做过 XX 从 5 千到 2 万的优化，思路是…"
              ② 进一步刺探："这个重构预计周期多久？团队目前人力够吗？"
  ```

**3. 回答引导**
- STAR 结构引导：反问环节不适用 STAR；改为"反问目的 + 预期信息 + 备选追问"三段引导。
- 关键追问点（此处为"候选人听回答后的二次应对"）：
  1. 听到挑战后，能否把自己的相关经历关联上？（展现匹配度）
  2. 能否根据面试官回答二次追问，深入刺探？（展现信息刺探能力）
  3. 避免问"加班多吗/年终奖多少"等只索取不展现的问题。
- 评分要点：
  - 优秀：反问落在三元交集，听回答后能关联自身经历并二次追问。
  - 合格：反问有针对性，能展现对职位的真实兴趣。
  - 不合格：问百度可查的问题，或只问福利待遇。

**4. 考察点**
- 主：社会角色层（职业成熟度）
- 次：动机层（对职位的真实兴趣）

**5. 难度等级**：不分级（反向提问题无难度梯度，依据 `question-taxonomy.md` §5 与 `boundaries.md` §5）

**6. 常见错误回答**
- 问百度可查的问题："公司主要做什么业务？"避坑：反问要刺探 JD 与公开资料查不到的信息。
- 只索取不展现："加班多吗？年终奖几个月？"避坑：反问要兼顾展现自己，不只问福利。
- 问太多：一次抛 5 个问题。避坑：反问 2-3 个即可，留时间听回答。

## 变体 6：八股文题模板

**模板差异**：参考答案以标准知识点为主（来自 RAG 题库，须标 `source` 与 `verified`）；关键追问点偏"为什么"（从 L1 记忆压向 L2 分析）；考察点主压知识层；难度多为 L1-L2。

### 完整示例

**1. 题目**
> 简述 TCP 三次握手的过程和每一步的作用。
- 溯源：`source` = 八股文题库（RAG 检索命中），`verified` = true；若题库未命中则 `verified` = false 并标注"未经题库验证"（见 `rag-hook-spec.md` fallback）

**2. 参考答案**
- 核心观点：三次握手是 TCP 建立可靠连接的过程，通过 SYN/SYN-ACK/ACK 三次报文交换，同步双方初始序列号并确认收发能力。
- 展开论述：
  1. 第一次握手：客户端发 SYN=1, seq=x，进入 SYN_SENT。作用：客户端告知服务端"我想建连"，同步客户端初始序列号 x。
  2. 第二次握手：服务端回 SYN=1, ACK=1, seq=y, ack=x+1，进入 SYN_RCVD。作用：服务端确认收到客户端 SYN，同时同步服务端初始序列号 y。
  3. 第三次握手：客户端发 ACK=1, seq=x+1, ack=y+1，双方进入 ESTABLISHED。作用：客户端确认收到服务端 SYN，连接建立，可双向传数据。
- 代码/示例（报文时序）：
  ```
  Client                              Server
    | --- SYN, seq=x ---------------> |  (第一次)
    | <--- SYN, ACK, seq=y, ack=x+1-- |  (第二次)
    | --- ACK, seq=x+1, ack=y+1 ----> |  (第三次)
    | ===== 双向数据传输 =========== |
  ```

**3. 回答引导**
- STAR 结构引导：八股文题不强制 STAR；改为"定义→步骤→每步作用→为什么这样设计"四段引导。
- 关键追问点（从 L1 记忆压向 L2 分析）：
  1. 记忆层：三次握手每一步的状态变化是什么？（L1）
  2. 分析层：为什么是三次而不是两次？两次会出现什么问题？（L2，考查可靠性理解）
  3. 深度层：如果第三次 ACK 丢了，连接会建立吗？服务端怎么处理？（L2-L3）
- 评分要点：
  - 优秀：能讲清三步状态变化，并能解释"为什么不是两次"（防历史报文/防双方资源浪费）。
  - 合格：能说清三步过程与每步作用。
  - 不合格：只背步骤说不出每步作用，或答不出状态变化。

**4. 考察点**
- 主：知识层（网络基础功底）
- 次：技能层（知识体系完整度，能否延伸到为什么）

**5. 难度等级**：L1（概念/定义，对应 P5，依据 `difficulty-grading.md`）

**6. 常见错误回答**
- 只背步骤不说作用："客户端发 SYN，服务端回 SYN+ACK，客户端再发 ACK。"避坑：每步要说清"为什么这一步"。
- 答不出"为什么不是两次"：被追问时卡壳。避坑：八股文准备要延伸到 Why 层。
- 混淆状态：把 SYN_RCVD 和 ESTABLISHED 搞混。避坑：状态机要记准。
