# 题型分类规则

> 本文件是 `SKILL.md` 的路由细则，定义 6 类面试题型的触发条件、生成规则、考察点与数量配比。
> 难度分级见 `difficulty-grading.md`；答案引导模板见 `answer-guidance-template.md`；边界见 `boundaries.md`。

## 总览

| 题型 | 主要素材来源 | 编排的 skill | 推荐配比 |
|------|-------------|-------------|---------|
| 1. 技术深挖题 | 简历技术栈 | highlight-extractor | 20%-25% |
| 2. 项目追问题 | 简历项目经历 | highlight-extractor + competency-model-alignment | 20%-25% |
| 3. 系统设计题 | JD 高并发/大规模要求 | jinsheng-san-yuanze | 15%-20% |
| 4. 行为面试题 | JD 软性要求 + 简历经历 | competency-model-alignment | 15%-20% |
| 5. 反向提问题 | JD 职位信息 | reverse-questioning-framework | 5%-10% |
| 6. 八股文题 | 技术栈基础知识 | RAG 外挂（见 `rag-hook-spec.md`） | 15%-20% |

配比说明：以上为单次题集（约 15-20 题）的推荐占比，可按 JD 偏重调整。技术岗偏重 1/2/6，管理岗偏重 3/4/5。总量上限受 token 预算约束（见 `boundaries.md`）。

## 1. 技术深挖题

**定义**：基于简历中写明的技术栈，深挖原理、源码理解或踩坑经验，验证技术的真实深度。

**触发条件**：
- 简历出现具体技术名词（如 React、Kafka、Redis、MySQL）。
- JD 要求该技术栈作为必备技能。
- 两者交集即为深挖靶点。

**生成规则**：
- 每个靶点技术生成 1-2 题，按"原理 → 源码 → 踩坑"递进。
- 题目必须可溯源到简历具体段落（标注 `resume_ref`）。
- 禁止出简历未提及的技术深挖题。
- 题目结构复现白海飞三层递进：表层事实（用过吗）→ 深度细节（原理/坑）→ 感受和观点（选型权衡）。

**考察点**：技能深度、学习能力、技术选型判断力（对应 competency-model-alignment 的技能层与潜力层）。

**数量配比建议**：占题集 20%-25%，约 3-5 题。

**示例题干模板**：
- 表层：你简历里提到用过 X，能简述它的核心机制吗？
- 深度：X 的底层是怎么实现 Y 的？你踩过最深的坑是什么？
- 观点：如果重新选型，你会在什么场景下不用 X 而用 Z？为什么？

## 2. 项目追问题

**定义**：基于简历项目经历，用 STAR 反向追问细节，验证项目真实性与个人贡献度。

**触发条件**：
- 简历含项目段落（项目名、角色、技术栈、成果）。
- JD 关注交付能力或复杂项目经验。

**生成规则**：
- 每个重点项目生成 2-3 个追问点，覆盖 STAR 四要素。
- 先用 `highlight-extractor` 榨取项目亮点作为追问靶点，再用 `competency-model-alignment` 按四层模型组织追问深度。
- 必须追问"个人贡献边界"（你做了哪部分，同事做了哪部分），防止团队成果冒充个人成果。
- 标注 `resume_ref` 溯源到简历项目段落。

**考察点**：经验真实性、个人贡献度、复杂问题解决能力、复盘能力（潜力层）。

**数量配比建议**：占题集 20%-25%，约 3-5 题。

**示例题干模板**（STAR 反向追问）：
- S/T：这个项目当时的背景和你承担的角色是什么？
- A：你具体做了哪部分？遇到的最大技术障碍是什么？怎么解决的？
- R：结果用什么指标衡量？如果重做你会怎么改？
- 贡献边界：刚才那些成果，有多少是你独立完成的？同事负责哪部分？

## 3. 系统设计题

**定义**：基于 JD 中高并发、大规模、高可用等要求，设计系统方案，验证架构能力。

**触发条件**：
- JD 出现"高并发""千万级用户""高可用""分布式""性能优化"等关键词。
- 目标职级 P6+（P6 开始要求独立当一面，P7+ 要求跨团队架构）。

**生成规则**：
- 先用 `jinsheng-san-yuanze` 的价值原则筛选 JD 中值得设计的高价值方向（能解决业务瓶颈的才算值得出题）。
- 题目给出业务场景 + 约束条件（QPS、数据量、一致性要求），要求候选人给出架构方案。
- 必须包含权衡点（如 CAP 取舍、成本 vs 性能），不出只有唯一标准答案的题。
- 标注 `jd_ref` 溯源到 JD 对应要求段落。

**考察点**：架构设计能力、技术选型权衡、业务理解、规模意识（对应 jinsheng-dicing-luoji 的下一级别能力要求）。

**数量配比建议**：占题集 15%-20%，约 2-4 题。初级岗（P5）可降为 0-1 题。

**示例题干模板**：
- 场景：JD 要求"设计支撑千万级日活的 XX 系统"，给定 QPS=10万、数据量=10亿条、强一致性要求。
- 追问：你的方案在 CAP 上怎么取舍？如果某机房挂了怎么办？成本能否压到预算内？

## 4. 行为面试题

**定义**：基于 JD 软性要求（协作、领导、抗压、冲突处理）+ 简历经历，设计行为面试题，考查素质模型下层（潜力与动机）。

**触发条件**：
- JD 出现"团队协作""带人""跨部门沟通""抗压""ownership"等软性要求。
- 目标职级 P6+（P7+ 强制要求带团队/跨团队协作）。

**生成规则**：
- 用 `competency-model-alignment` 的四层模型（经验-技能-潜力-动机）设计，重点压向下层。
- 题目用"请举一个具体例子"开头，强制 STAR 回答，禁止表态式回答。
- 基于 BEI（行为事件访谈）方法：通过对过去行为具体细节的深度挖掘预测未来表现。
- 标注 `jd_ref`（软性要求）+ `resume_ref`（可关联的经历，若有）。

**考察点**：协作能力、领导力、抗压能力、价值观与动机（素质模型下层）。

**数量配比建议**：占题集 15%-20%，约 2-4 题。管理岗可上调至 25%。

**示例题干模板**（BEI 行为题）：
- 协作：请举一个你和同事在技术方案上产生严重分歧的例子，最后怎么解决的？
- 领导：请讲一次你主动带领团队攻克难题的经历，你的具体动作是什么？
- 抗压：请描述你压力最大的一段项目经历，你怎么撑过来的？
- 动机：为什么当初你要主动接手那个没人愿意做的烂摊子？

## 5. 反向提问题

**定义**：候选人向面试官提问的环节，编排 `reverse-questioning-framework` 生成高质量反问。

**触发条件**：
- 题集需覆盖面试全流程时必含此类。
- 用户明确想准备"你有什么想问我的"环节。

**生成规则**：
- 直接调用 `reverse-questioning-framework` 的三元交集模型（应聘者需求 ∩ 面试官需求 ∩ 职位要求）。
- 基于 JD 信息生成 2-3 个反问，每个反问标注其落在三元交集的哪个区。
- 标注 `jd_ref` 溯源到 JD 中可刺探的信息点。

**考察点**：职业成熟度、对职位的真实兴趣、信息刺探能力。

**数量配比建议**：占题集 5%-10%，约 1-2 题（反问通常 2-3 个即可）。

**示例题干模板**：
- "请问团队目前面临的最大技术挑战是什么？如果我有幸加入，您希望我能在哪些方面快速贡献？"（落在：应聘者关心成长 ∩ 面试官关心能否干活 ∩ 职位交付挑战）

## 6. 八股文题

**定义**：基础知识题，RAG 外挂题库检索，验证技术基本功。

**触发条件**：
- JD 要求的技术栈有标准基础知识考点（如 TCP、HTTP、JVM、数据库索引、并发原语）。
- 题库已有该技术栈的题目（命中规则见 `rag-hook-spec.md`）。

**生成规则**：
- 调用 RAG 外挂接口：输入（技术栈 + 难度 + 数量）→ 输出（题目 + 答案 + 标签）。
- 接口契约与 fallback 策略见 `rag-hook-spec.md`。
- 题库未命中时 AI 生成并标记 `verified: false`（未经题库验证）。
- 难度按 `difficulty-grading.md` 的 L1-L2 为主（八股文多为基础概念，深度题归入技术深挖题）。

**考察点**：基础功底、知识体系完整度。

**数量配比建议**：占题集 15%-20%，约 3-4 题。纯技术岗可上调，管理岗下调。

**示例题干模板**：
- L1：简述 TCP 三次握手的过程和每一步的作用。
- L2：TCP 三次握手中，为什么不是两次或四次？请从可靠性角度分析。

## 题型选择决策表

| 输入信号 | 优先题型 |
|---------|---------|
| 简历技术栈丰富，JD 要求同栈 | 技术深挖题 + 八股文题 |
| 简历有重点项目 | 项目追问题 |
| JD 含高并发/大规模 | 系统设计题 |
| JD 含协作/领导/抗压 | 行为面试题 |
| 用户要准备全流程 | 全部 6 类按配比 |
| 用户只要反问准备 | 仅反向提问题（直接走 reverse-questioning-framework） |
| 简历与 JD 有能力缺口 | 追加 blind-spot-navigation 盲区降维预案 |
